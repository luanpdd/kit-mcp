### Task 3.1 — `supabase-edge-functions/SKILL.md`

**Path:** `D:/projetos/opensource/mcp/kit/skills/supabase-edge-functions/SKILL.md`

**Frontmatter:**
- `name: supabase-edge-functions`
- `description: Use ao escrever Edge Functions — Deno + imports npm:/jsr: (NUNCA bare), Deno.serve, env vars pre-populadas, file writes APENAS em /tmp, EdgeRuntime.waitUntil.` (≤ 200 chars)

**Must-include (verbatim):** `npm:`, `jsr:`, `Deno.serve`, `EdgeRuntime.waitUntil`, `/tmp`

**Conteúdo concreto obrigatório:**
- Regra: runtime é **Deno**, não Node.js — usar APIs Deno (`Deno.serve`, `Deno.env`)
- Regra: imports SEMPRE com `npm:` ou `jsr:` prefix — NUNCA bare specifiers (`import x from 'pkg'` quebra)
- Regra: env vars pre-populadas pelo runtime: `SUPABASE_URL`, `SUPABASE_PUBLISHABLE_KEYS`, `SUPABASE_SECRET_KEYS` — não definir manualmente
- Regra: `Deno.serve` é o entry point canônico (não `addEventListener('fetch')`)
- Regra: file writes APENAS em `/tmp` — qualquer outro path é read-only
- Regra: para tarefas em background após resposta, usar `EdgeRuntime.waitUntil(promise)` — sem isso, função termina antes da promise
- Regra: multi-rota com Hono ou Express deve prefixar todas as rotas com `/<function-name>` (ex: `/my-function/users`)
- Pattern canônico — função básica (code block EN):
  ```ts
  // supabase/functions/hello/index.ts
  // PT-BR: imports versionados sempre com npm:/jsr:
  import { createClient } from 'npm:@supabase/supabase-js@2'

  Deno.serve(async (req) => {
    const supabase = createClient(
      Deno.env.get('SUPABASE_URL')!,
      Deno.env.get('SUPABASE_SECRET_KEYS')!
    )
    const { data } = await supabase.from('tasks').select('*').limit(10)
    return new Response(JSON.stringify(data), {
      headers: { 'Content-Type': 'application/json' }
    })
  })
  ```
- Pattern canônico — background task com `waitUntil`:
  ```ts
  Deno.serve(async (req) => {
    const body = await req.json()
    // PT-BR: trabalho leve responde rápido; pesado roda em background
    EdgeRuntime.waitUntil((async () => {
      await Deno.writeTextFile('/tmp/audit.log', JSON.stringify(body))
      // PT-BR: chamadas a APIs externas, geração de embeddings, etc.
    })())
    return new Response('accepted', { status: 202 })
  })
  ```
- Pattern canônico — multi-rota com Hono:
  ```ts
  // supabase/functions/api/index.ts
  import { Hono } from 'npm:hono'
  const app = new Hono().basePath('/api')
  app.get('/users', (c) => c.json({ users: [] }))
  app.get('/users/:id', (c) => c.json({ id: c.req.param('id') }))
  Deno.serve(app.fetch)
  ```
- Anti-pattern (Errado/Por quê/Certo):
  - **Errado:** `import { foo } from '@supabase/supabase-js'`
  - **Por quê:** bare specifier não resolve em Deno — runtime falha em startup
  - **Certo:** `import { foo } from 'npm:@supabase/supabase-js@2'`
- Anti-pattern: `Deno.writeTextFile('/data/x.log', ...)` → APENAS `/tmp` é writable
- Anti-pattern: trabalho pesado inline na response → usar `EdgeRuntime.waitUntil` para liberar resposta
- Anti-pattern: multi-rota sem prefix `/<function-name>` → request 404 quando deployado

### Task 3.2 — `supabase-storage/SKILL.md`

**Path:** `D:/projetos/opensource/mcp/kit/skills/supabase-storage/SKILL.md`

**Frontmatter:**
- `name: supabase-storage`
- `description: Use ao integrar Storage — buckets públicos vs privados, signed URLs com expiration, RLS sobre storage.objects com multi-tenant path, image transforms, TUS uploads.` (≤ 200 chars)

**Must-include (verbatim):** `signed URL`, `storage.objects`, `multi-tenant path`

**Conteúdo concreto obrigatório:**
- Regra: distinguir buckets **públicos** (URL direta, CDN cache) vs **privados** (signed URL com expiration)
- Regra: signed URLs com `expiresIn` curto (ex: 60s para downloads, 3600s para imagens)
- Regra: RLS sempre habilitada em `storage.objects` — sem RLS, qualquer authenticated lê qualquer bucket privado
- Regra: **multi-tenant path isolation** — usar `auth.uid()` como path prefix (`<user_id>/<filename>`) e validar em RLS via `(storage.foldername(name))[1] = (select auth.uid())::text`
- Regra: image transformations apenas em buckets com transformation enabled — `?width=800&height=600`
- Regra: para uploads grandes (> 6 MB), usar TUS resumable protocol (`@supabase/storage-js` com `uploadToSignedUrl`)
- Regra: awareness de **egress billing** — bucket público sem cache headers customizados pode disparar custo
- Pattern canônico — RLS multi-tenant (code block EN):
  ```sql
  -- PT-BR: usuário só lê arquivos sob seu próprio path prefix
  create policy "users_read_own_files"
    on storage.objects for select to authenticated
    using (
      bucket_id = 'private-uploads'
      and (storage.foldername(name))[1] = (select auth.uid())::text
    );

  create policy "users_insert_own_files"
    on storage.objects for insert to authenticated
    with check (
      bucket_id = 'private-uploads'
      and (storage.foldername(name))[1] = (select auth.uid())::text
    );
  ```
- Pattern canônico — signed URL (code block EN):
  ```ts
  // PT-BR: signed URL com expiração 1h
  const { data, error } = await supabase
    .storage
    .from('private-uploads')
    .createSignedUrl(`${userId}/avatar.jpg`, 3600)
  ```
- Pattern canônico — upload com path multi-tenant:
  ```ts
  const { data: { user } } = await supabase.auth.getUser()
  const path = `${user!.id}/avatar.jpg`
  await supabase.storage.from('private-uploads').upload(path, file)
  ```
- Anti-pattern (Errado/Por quê/Certo):
  - **Errado:** `await supabase.storage.from('uploads').upload('avatar.jpg', file)`
  - **Por quê:** path sem prefix de tenant — qualquer usuário sobrescreve `avatar.jpg` global
  - **Certo:** `upload(\`${user.id}/avatar.jpg\`, file)`
- Anti-pattern: bucket privado sem RLS em `storage.objects` → qualquer authenticated lê tudo
- Anti-pattern: overwrite de arquivo público com mesmo nome → CDN cache stale; use versionamento ou path único
- Anti-pattern: `getPublicUrl` em bucket privado → URL não funciona; usar `createSignedUrl`

**Note (futuro):** Vector Buckets / Analytics Buckets ainda alpha em 2026-05-06; mencionar como existência mas não detalhar (risco de breaking).

### Task 3.3 — `supabase-pgvector-rag/SKILL.md`

**Path:** `D:/projetos/opensource/mcp/kit/skills/supabase-pgvector-rag/SKILL.md`

**Frontmatter:**
- `name: supabase-pgvector-rag`
- `description: Use ao implementar RAG com pgvector — create extension vector, dim consistente, HNSW vs IVFFlat, operadores <=>/<#>, RAG with permissions via RLS, chunking.` (≤ 200 chars)

**Must-include (verbatim):** `HNSW`, `IVFFlat`, `<=>`, `RAG with permissions`

**Conteúdo concreto obrigatório:**
- Regra: setup com `create extension if not exists vector;`
- Regra: definir **dimension fixa** por modelo (ex: 1536 para OpenAI ada-002, 768 para nomic-embed-text)
- Regra: escolher index — **HNSW** = recall melhor + queries mais rápidas com mais data; **IVFFlat** = build mais rápido mas recall menor (defaultar HNSW)
- Regra: distance operators canônicos — `<=>` cosine distance (mais comum), `<#>` negative inner product, `<->` L2
- Regra: **RAG with permissions** — combinar similarity search com RLS na tabela source para que retrieval respeite permissões do usuário (anti-pitfall A11)
- Regra: chunking — chunks de 200-500 tokens com overlap 10-20% (chunks muito grandes degradam embeddings; muito pequenos perdem contexto)
- Pattern canônico — schema + index HNSW (code block EN):
  ```sql
  create extension if not exists vector;

  create table public.documents (
    id uuid primary key default gen_random_uuid(),
    user_id uuid not null references auth.users(id),
    content text not null,
    embedding vector(1536) not null,
    created_at timestamptz default now()
  );

  -- PT-BR: HNSW recall melhor; cosine distance
  create index on public.documents using hnsw (embedding vector_cosine_ops);

  alter table public.documents enable row level security;
  create policy "users_read_own_docs"
    on public.documents for select to authenticated
    using ((select auth.uid()) = user_id);
  ```
- Pattern canônico — RAG with permissions (function):
  ```sql
  create or replace function public.match_documents(
    query_embedding vector(1536),
    match_threshold float,
    match_count int
  )
  returns table (id uuid, content text, similarity float)
  language plpgsql security invoker set search_path = ''
  stable
  as $$
  begin
    return query
      select
        d.id,
        d.content,
        1 - (d.embedding <=> query_embedding) as similarity
      from public.documents d
      where 1 - (d.embedding <=> query_embedding) > match_threshold
      order by d.embedding <=> query_embedding
      limit match_count;
  end;
  $$;
  ```
- Pattern canônico — IVFFlat alternativa (quando build time importa):
  ```sql
  create index on public.documents using ivfflat (embedding vector_cosine_ops) with (lists = 100);
  ```
- Anti-pattern (Errado/Por quê/Certo):
  - **Errado:** `embedding vector(1536)` na tabela mas embedder retorna 768 dim
  - **Por quê:** dim mismatch — insert falha silenciosamente ou produz match aleatório
  - **Certo:** dim da coluna casa com modelo; tipos verificados em CI
- Anti-pattern: similarity search sem RLS na tabela source → usuário recupera documentos de outros (vazamento via RAG)
- Anti-pattern: chunks de 5k tokens → embedding perde detalhe semântico; usar 200-500 com overlap

**Note (futuro):** hybrid search com FTS + RRF está em `supabase-fts` (defer v1.9).

### Task 3.4 — `supabase-cron-queues/SKILL.md`

**Path:** `D:/projetos/opensource/mcp/kit/skills/supabase-cron-queues/SKILL.md`

**Frontmatter:**
- `name: supabase-cron-queues`
- `description: Use ao orquestrar background jobs — pg_cron + pgmq + pg_net pattern cron → pgmq → Edge Function. Sem dependência externa. Postgres 15.6.1.143+.` (≤ 200 chars)

**Must-include (verbatim):** `pg_cron`, `pgmq`, `pg_net`, `cron → pgmq → Edge Function`

**Conteúdo concreto obrigatório:**
- Regra: extensions necessárias — `pg_cron` (jobs scheduled), `pgmq` (Postgres Message Queue, requer Postgres 15.6.1.143+), `pg_net` v0.10.0+ (HTTP do banco)
- Regra: **pattern canônico** `cron → pgmq → Edge Function` — pg_cron enfileira em pgmq, Edge Function consome via `pg_net.http_post` ou polling
- Regra: jobs `pg_cron` devem ser **curtos** (< 10 min) — jobs longos bloqueiam scheduler; despachar para Edge Function via queue
- Regra: usar `pgmq.send` para enfileirar e `pgmq.read` + `pgmq.archive` para consumir (visibility timeout previne double processing)
- Regra: `pg_net` async — `net.http_post` retorna `request_id`, response chega em `net._http_response`
- Pattern canônico — setup das extensions (code block EN):
  ```sql
  create extension if not exists pg_cron;
  create extension if not exists pgmq;
  create extension if not exists pg_net;

  -- PT-BR: criar fila
  select pgmq.create('email_jobs');
  ```
- Pattern canônico — `cron → pgmq → Edge Function`:
  ```sql
  -- PT-BR: cron job a cada 5 min enfileira mensagens
  select cron.schedule(
    'enqueue-pending-emails',
    '*/5 * * * *',
    $$
    insert into pgmq.q_email_jobs (message)
    select jsonb_build_object('user_id', id, 'kind', 'reminder')
    from public.users
    where pending_email = true;
    $$
  );

  -- PT-BR: cron job a cada minuto despara processamento via Edge Function
  select cron.schedule(
    'process-email-queue',
    '*/1 * * * *',
    $$
    select net.http_post(
      url := 'https://<project-ref>.supabase.co/functions/v1/process-emails',
      headers := jsonb_build_object('Authorization', 'Bearer ' || current_setting('supabase.functions_token'))
    );
    $$
  );
  ```
- Pattern canônico — Edge Function consumer:
  ```ts
  // supabase/functions/process-emails/index.ts
  Deno.serve(async () => {
    const supabase = createClient(
      Deno.env.get('SUPABASE_URL')!,
      Deno.env.get('SUPABASE_SECRET_KEYS')!
    )
    // PT-BR: pgmq.read pega N mensagens com visibility timeout
    const { data: msgs } = await supabase.rpc('pgmq_read', {
      queue_name: 'email_jobs',
      vt: 30,
      qty: 10
    })
    for (const m of msgs ?? []) {
      // PT-BR: processar mensagem
      await sendEmail(m.message)
      await supabase.rpc('pgmq_archive', { queue_name: 'email_jobs', msg_id: m.msg_id })
    }
    return new Response('ok')
  })
  ```
- Anti-pattern (Errado/Por quê/Certo):
  - **Errado:** `select cron.schedule('heavy-job', '0 * * * *', $$ select pg_sleep(900); ... $$);`
  - **Por quê:** jobs > 10 min em pg_cron bloqueiam scheduler + falha em retry
  - **Certo:** cron enfileira em pgmq + Edge Function processa
- Anti-pattern: chamar API externa direto de pg_cron via `pg_net` síncrono → usar pattern enqueue + Edge Function consumer
- Anti-pattern: pgmq.read sem `pgmq.archive` ou `pgmq.delete` → mensagem reaparece após visibility timeout (double processing)
