# RUNBOOK — kit-mcp Emergency Response Guide

> Operational playbook for the maintainer when something breaks at runtime. Each scenario follows the canonical **Symptom → Diagnosis → Fix** shape from the [`blameless-postmortems`](../kit/skills/blameless-postmortems/SKILL.md) and [`production-readiness-review`](../kit/skills/production-readiness-review/SKILL.md) skills.
>
> **Audience:** kit-mcp maintainers (single-repo, single-human today) plus contributors triaging an issue. **Cross-references:** SLOs at [`./slos/`](./slos/) for "is the service degraded?" decisions; metrics module at [`src/core/metrics.js`](../src/core/metrics.js) for live signal readout via the `metrics-snapshot` MCP tool.
>
> Generated by Phase 96 (v1.18 — Eat Your Own Dog Food).

## 1. MCP server boot fail

**Symptom**

- IDE (Claude Code, Cursor, Codex, Gemini, Windsurf, Antigravity, Copilot, Trae) reports the MCP server `kit-mcp` failed to start, errored on initialization, or never produces a `tools/list` response.
- Running `node bin/mcp.js` directly produces no output on stdout, or exits non-zero with a stack on stderr.
- New install path: `npx -y @luanpdd/kit-mcp install <ide>` succeeded but the IDE still shows the server as broken on next launch.

**Diagnosis**

Run, in order:

```bash
# 1. Capture a clean stdio handshake. Should print a single JSON-RPC banner
#    line on stdout within ~500ms when working.
node bin/mcp.js < /dev/null 2>&1 | head -20

# 2. Check Node version. The package.json `engines.node` is "20 || 22".
node --version

# 3. Sanity-check dependencies. A botched install of `@modelcontextprotocol/sdk`,
#    `commander`, or `chokidar` (the only 3 runtime deps) is the most common
#    cause of a thrown exception at module load.
npm ci --omit=optional

# 4. If running globally, locate the installed binary and check it executes.
which kit-mcp && file "$(which kit-mcp)"

# 5. Run the project doctor — covers version mismatch, settings.json, lockfile,
#    and `.planning/` directory health in one shot.
node bin/cli.js doctor --project-root .
```

**Fix**

| Diagnosis | Fix |
|---|---|
| Node < 20 | Upgrade Node to 20 or 22 LTS. The package refuses to start on older runtimes. |
| `npm ci` errors out | Remove `node_modules` and `package-lock.json`, then `npm install`. The 3-deps + 3-optional-deps budget is small enough that a clean reinstall is cheap. |
| `npx -y @luanpdd/kit-mcp@latest --version` mismatches `package.json` | The IDE has a stale path cached. Re-run `kit install <ide>` and restart the IDE. |
| Stack trace mentions `Cannot find module '@modelcontextprotocol/sdk'` | The user installed without optional deps. Run `npm install` (not `--omit=optional`); the SDK is a *required* dep, not optional. |
| Doctor flags `.planning/STATE.md` malformed | Restore from git (`git checkout .planning/STATE.md`) or run `node ./.claude/framework/bin/tools.cjs state init`. |
| Stack trace mentions `permission denied` opening sidecar lockfile | See [Scenario 2](#2-sidecar-ui-hangunresponsive) — the boot pre-checks lockfile health. |
| Nothing helps | File a [GitHub issue](https://github.com/luanpdd/kit-mcp/issues/new) with the output of `node bin/cli.js doctor --json` attached. |

**Verification**

```bash
# Drive a single tools/list request and confirm the server responds.
echo '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2024-11-05","capabilities":{},"clientInfo":{"name":"runbook","version":"0"}}}' \
  | node bin/mcp.js | head -3
# Expected: a JSON line containing `"result":{"protocolVersion":"2024-11-05",...`
```

## 2. Sidecar UI hang/unresponsive

**Symptom**

- `kit ui status` does not return within ~2 seconds.
- `kit ui open` reports "no sidecar running" but a stale lockfile exists in the project's `.claude/` or `.kit-mcp/` state directory.
- The browser tab at `http://localhost:7100` (or 7101–7199 if the auto-port escalation kicked in) hangs on load.
- Hooks fire from the LLM client but no events appear in the UI.

**Diagnosis**

```bash
# 1. Check whether a process actually owns the lockfile.
node bin/cli.js ui status --project-root .
# Stale lockfile → "no sidecar running" but the file still exists on disk.

# 2. Inspect the lockfile contents directly.
cat .claude/.kit-mcp/sidecar.lock 2>/dev/null || cat .kit-mcp/sidecar.lock 2>/dev/null
# Fields: pid, port, startedAt, ttlSec.

# 3. Check the port range. Sidecar binds 7100 first, escalates to 7101..7199.
ss -tlnp 2>/dev/null | grep -E '710[0-9]|7[12][0-9]{2}' || netstat -an | grep 71

# 4. Probe the healthz endpoint of the suspected port.
curl -fsS http://localhost:7100/healthz | head -5

# 5. The Phase v1.12.1 race fix (commit 56b327f) ensures the hook's TCP write
#    completes before the parent exits. If you suspect the race regressed,
#    check the most recent hook fire produced an event in the UI.
```

**Fix**

| Diagnosis | Fix |
|---|---|
| Stale lockfile (PID not running) | `kit ui stop --project-root .` is idempotent — it removes a stale lockfile if no process responds to `/shutdown`. Then restart with `kit ui start` or trigger auto-spawn by running any `kit` command. |
| Port 7100 occupied by foreign process | The auto-port logic should have escalated; if not, set `KIT_MCP_UI_PORT_BASE=7200` in your shell and restart. |
| Sidecar process is alive but `/healthz` hangs | Kill the process: read PID from lockfile, `kill <pid>`. Then `kit ui stop` to clear the lockfile, then `kit ui start`. |
| Hooks fire but UI shows nothing | Race regression suspected. Reproduce with `KIT_MCP_HOOK_TRACE=1 <run-anything>` and capture the stderr trace; the v1.12.1 fix should keep `flush + exit` ordered. File an issue with the trace. |
| Browser tab loads but `Connection lost` keeps appearing | SSE keepalive failing — check for a proxy/VPN intercepting localhost. The sidecar writes a keepalive every 25s. |

**Verification**

```bash
# UI is healthy if all three of these succeed:
kit ui status --project-root .                                   # → "🟢 running on http://localhost:71xx"
curl -fsS http://localhost:71xx/healthz                          # → 200 OK
curl -fsS -N http://localhost:71xx/events --max-time 2 | head -3  # → SSE stream opens, optional event lines
```

## 3. Manifest mismatch on sync

**Symptom**

- `kit sync install <target>` fails with an error envelope where `code === 'EMANIFESTMISMATCH'`.
- The CLI prints something like: `manifest mismatch: <file> expected sha=<a>, got sha=<b>`.
- The MCP server returns the same `EMANIFESTMISMATCH` code via the sanitized envelope (Phase 84 redaction preserves the code, strips paths/stack).

**Diagnosis**

The manifest is the SHA-256 fingerprint over the canonical `kit/` tree, regenerated at `prepublish`. A mismatch means **the on-disk kit content drifted from the manifest the publisher signed.** Three legitimate causes plus one attack signal:

```bash
# 1. Did you edit kit/ locally? If yes, the manifest is stale on YOUR machine
#    until you regenerate it. This is the dominant case for contributors.
git status kit/

# 2. Did npm install corrupt a file? Rare but possible on systems with
#    aggressive antivirus / Windows Defender quarantining .md content.
node -e "import('./src/core/manifest-verify.js').then(m => m.verifyManifest('./kit')).then(r => console.log(r))"

# 3. Are you running a forked kit (custom personal kit) without regenerating
#    the manifest?  This is expected; treat it like cause #1.

# 4. ATTACK SIGNAL: if none of the above apply and you didn't touch kit/,
#    the published tarball may have been tampered. Compare the on-disk
#    kit/MANIFEST.json against the GitHub release artifact.
```

**Fix**

| Diagnosis | Fix |
|---|---|
| You edited `kit/` and the manifest is stale (contributor flow) | Run `npm run regen-manifest` (defined in `package.json` `scripts`) to rewrite `kit/MANIFEST.json` against the current tree, then commit and re-run sync. |
| Local file corrupted by a third-party tool | Restore the file from git: `git checkout kit/<file>`. Re-run sync. |
| Forked kit / custom kit_root | This is the expected dev path. Either set `KIT_MCP_SKIP_MANIFEST_CHECK=1` in your shell and re-run sync (**dev environments only — never set this in production / shared CI**), or run `npm run regen-manifest` against your fork and commit. |
| Manifest tampered (attack signal) | Stop. Do not run sync. Compare `kit/MANIFEST.json` against the GitHub release tarball for the version reported by `npx @luanpdd/kit-mcp --version`. If they differ, file a security issue and uninstall. |

**Verification**

```bash
# Regenerate locally and confirm clean run.
npm run regen-manifest && \
  node bin/cli.js sync install claude-code --project-root /tmp/sync-test
# Expected: "Total: <N>" line, no EMANIFESTMISMATCH.
```

> **Cross-ref:** Phase 83 introduced the `verifyManifest` guard; Phase 90.01 added an in-memory cache that only retains `ok=true` results so a re-run after fix recovers immediately. See [`.planning/phases/90-perf-runtime-wave-2/`](./phases/90-perf-runtime-wave-2/) and [`src/core/manifest-verify.js`](../src/core/manifest-verify.js).

## 4. npm publish workflow fail

**Symptom**

- `git push origin v<X.Y.Z>` succeeded, but the GitHub Actions release workflow shows red.
- `npm view @luanpdd/kit-mcp version` still shows the previous version.
- A user reports `npm install @luanpdd/kit-mcp@<X.Y.Z>` returns 404.

**Diagnosis**

```bash
# 1. Inspect the failing workflow run.
gh run list --workflow release.yml --limit 5
gh run view <run-id> --log-failed | head -100

# 2. Common failure modes (in observed frequency order):
#    a) CHANGELOG.md missing the new version section (release-please / changelog gate)
#    b) package.json version not matching the tag (e.g. tagged v1.18.0 but package still 1.17.x)
#    c) `npm audit` newly flags a transitive dep as critical (audit gate, Phase 92.01)
#    d) NPM_TOKEN expired or revoked
#    e) Manifest regeneration failed because kit/ has a non-canonical file name

# 3. If publish ran but uploaded a broken tarball, compare:
npm pack --dry-run  # files this would publish locally
# vs the failed workflow's `npm pack` output in the run log.
```

**Fix**

| Diagnosis | Fix |
|---|---|
| CHANGELOG.md missing entry | Add the version section, commit (`docs(changelog): v<X.Y.Z> notes`), bump tag (`git tag -d v<X.Y.Z> && git push --delete origin v<X.Y.Z> && git tag v<X.Y.Z> && git push --tags`). The workflow re-fires on tag. |
| `package.json` version mismatch | Bump `version` field, commit, retag. Same retag flow as above. |
| `npm audit` critical | Pin or upgrade the offending dep, commit, retag. **Do not** add `--audit-level high` to bypass — the gate is intentional. |
| NPM_TOKEN problem | Mint a new token at npmjs.com (Automation token), update the GitHub Actions secret `NPM_TOKEN`, retag. |
| Manifest regen fails | Run `npm run regen-manifest` locally, commit the output, retag. |
| Workflow succeeded but `npm view` lags | npm cache propagation can take 1–2 minutes. Wait, then re-check. |

**Verification**

```bash
# After workflow turns green:
gh run view <run-id>                     # status: success
npm view @luanpdd/kit-mcp version        # equals tagged version
npm view @luanpdd/kit-mcp dist.tarball   # download URL exists
curl -fsSI "$(npm view @luanpdd/kit-mcp dist.tarball)"  # 200 OK
```

> **Pipeline simplicity:** The hot path is literally `git push --tags → workflow runs → npm publish`. There is no staging registry, no canary cohort. The audit gate is the only defense, intentionally; see [`hermetic-builds`](../kit/skills/hermetic-builds/SKILL.md) for why we picked simplicity over multi-stage rollout.

## 5. Sync corruption (partial write)

**Symptom**

- After a `sync install` or hook-triggered re-sync, files are missing in the destination `.claude/` (or `.cursor/`, `.codex/`, etc.).
- Some files exist but are 0 bytes or truncated.
- Re-running sync produces the same partial state, suggesting it is not a transient OS hiccup.

**Diagnosis**

```bash
# 1. Re-run sync in dry-run mode to see the planned op set.
node bin/cli.js sync install <target> --project-root . --dry-run

# 2. Compare planned ops against actual destination tree.
find .claude -type f | wc -l          # actual file count
# vs the "Total: N" reported by the dry-run.

# 3. Check for permission errors that the previous sync swallowed.
node bin/cli.js sync install <target> --project-root . 2>&1 | grep -iE 'eacces|eperm|enospc'

# 4. Disk space sanity check — kit-mcp does not pre-check, write fails late.
df -h .

# 5. Inspect the previous sync's manifest snapshot (Phase 91 written[] list).
ls -la .claude/.kit-mcp/last-sync.json 2>/dev/null
```

**Fix**

| Diagnosis | Fix |
|---|---|
| 0-byte files from interrupted write | `kit sync remove <target> --project-root .` then `kit sync install <target> --project-root .`. Sync is idempotent and restartable. |
| `EACCES`/`EPERM` on a file | The destination directory contains a file the user has no write permission for (common with WSL ↔ Windows path shadowing). Move/rename the offending file, retry. |
| `ENOSPC` (out of disk space) | Free space, retry. The kit takes ~3.6 MB unpacked; sidecar logs and `.planning/` can grow large independently. |
| Sync says "Total: 321" but only ~100 files exist | A previous run was killed mid-transfer leaving phantom dirs. `kit sync remove` clears them; retry install. |
| Files appear briefly then vanish | Antivirus quarantine. Whitelist the destination tree, retry. |

**Verification**

```bash
# Healthy sync state:
kit sync status <target> --project-root .                # all rows ✓
diff -r kit .claude/agents 2>&1 | head                   # destination matches kit
test -f .claude/.kit-mcp/last-sync.json && echo "manifest snapshot present"
```

> **Cross-ref:** Phase 91 added a diff filter so re-running sync after a partial write actually re-writes the missing files (mtime+size heuristic surfaces "src newer than dst"). The `written[]` return contract is preserved (Phase 91 decision) — `Total: N` lists all op paths, not just files actually re-written.

## 6. CI coverage gate regression

**Symptom**

- A PR or push to `main` fails the line-coverage gate with `::error::line coverage XX% < 86% threshold`.
- The local `npm test` passed, but the CI step `Audit — line coverage threshold` exits non-zero.
- The threshold of `86` was introduced in Phase 100.02 (v1.20 ratchet — see [`ci.yml`](../.github/workflows/ci.yml) `THRESHOLD=86` block) and applies on `ubuntu-latest` × Node 22 × `claude-code` only.

**Diagnosis**

```bash
# 1. Reproduce the CI coverage run locally — the "%" is reported on the
#    "ℹ all files" footer of the Node 22 built-in coverage reporter.
node --experimental-test-coverage --test test/unit/*.test.js > /tmp/cov.txt 2>&1
grep "^ℹ all files" /tmp/cov.txt

# 2. List per-file coverage sorted ascending — the regressed file is usually
#    one that was edited in the same PR and lost test coverage.
grep "^ℹ src/" /tmp/cov.txt | sort -t'|' -k2 -n | head -10

# 3. If the regression is in a file the PR did not touch, somebody removed
#    or skipped a test. Bisect with git:
git log -p --since="1 week ago" -- test/unit/ | head -200

# 4. If the threshold itself feels wrong (paths were extracted/refactored
#    legitimately), inspect the threshold history block in ci.yml — it
#    documents the 65 → 75 → 80 → 86 ratchet path with rationale.
grep -n "THRESHOLD" .github/workflows/ci.yml
```

**Fix**

| Diagnosis | Fix |
|---|---|
| File touched in PR added uncovered lines | Add targeted unit tests covering the new branches. The pattern from Phase 100.01 is one test file per hot path (`render-paths.test.js`, `core-ui-paths.test.js`, etc.) — keep the same one-file-per-target shape. |
| Test removed by mistake (`git log -p test/unit/` shows the deletion) | Revert the deletion or add a replacement test that preserves the assertion intent. |
| Refactor legitimately moved lines into harder-to-test territory (live spawn, interactive TTY, network) | Either extract the testable helper into a sibling module (`*-paths.js`) and import it, or document an inline rationale and **temporarily** rollback the `THRESHOLD` value in `ci.yml`. Open a follow-up issue with the v1.21+ ratchet path (see Phase 100.02 SUMMARY for the canonical 3 avenues — mutation gate, helper extraction, branch coverage). |
| New `__test`-only export added to `src/` | Reject — Stable API v1.0+ forbids new `__test` exports. Use SDK-internal access pattern (`server._requestHandlers`) or fixture-driven end-to-end tests instead, per Plan 100-01 decision 1. |
| `cli/index.js` is the cap (currently 82.61% — see ci.yml history block) | Expected. Don't try to lift it without extracting helpers; the uncovered paths require live spawn or interactive TTY by design. Lifting that file is queued for v1.21+ (Phase 105 follow-up). |

**Verification**

```bash
# Re-run the same one-shot the CI step uses — exit 0 when the gate passes.
node --experimental-test-coverage --test test/unit/*.test.js > /tmp/cov.txt 2>&1
LINE_COV=$(grep "^ℹ all files" /tmp/cov.txt | head -1 | awk -F'|' '{print $2}' | tr -d ' %')
echo "Line coverage: $LINE_COV% (threshold: 86%)"
[ "${LINE_COV%.*}" -ge 86 ] && echo "OK" || echo "STILL BELOW THRESHOLD"
```

> **Cross-ref:** Phase 100 (Coverage ratchet 80 → 86 with rationale + 3 avenues for v1.21+) at [`.planning/phases/100-coverage-ratchet-80-90/`](./phases/100-coverage-ratchet-80-90/). The exhaustive threshold history (65 → 75 → 80 → 86) lives inline in `ci.yml` so future maintainers don't need to leave the file to understand the ratchet path.

## 7. Auto-snapshot persist failure

**Symptom**

- The MCP server stderr stream shows `[kit-mcp] auto-snapshot persist failed: <error>` repeatedly (once per `metrics-snapshot` tool call after the 1s throttle window).
- Tool handler still returns the in-memory payload normally — the contract `{counters, latency}` is preserved (Phase 102 graceful design).
- But `.planning/metrics/snapshots/` is empty or stale, and `/burn-rate-status` reports `no_data` because [`metrics.loadSnapshots`](../src/core/metrics.js) finds nothing on disk to read.

**Diagnosis**

```bash
# 1. Capture the exact error type from stderr — common categories are
#    ENOSPC (disk full), EACCES (permission denied), EISDIR / ENOTDIR
#    (path-shape mismatch — usually .planning/metrics existing as a regular
#    file), and EROFS (read-only mount).
# Filter stderr from the IDE's MCP log if available, or reproduce by calling
# the tool directly via your MCP client and watching stderr.

# 2. Disk space — kit-mcp writes JSON snapshots ~5-15 KB each, plus rolling
#    cleanup of files older than 30d (Phase 99). If /home or %USERPROFILE%
#    is at 100%, the persist fails even though the in-memory payload returns.
df -h .planning

# 3. Permissions — the snapshots subdir must be writable by the same user
#    that runs the MCP server (the IDE process owner).
ls -la .planning/metrics/snapshots/ 2>/dev/null || echo "directory missing"

# 4. Path-shape — if .planning/metrics exists as a regular FILE rather than
#    a directory, mkdir(.../snapshots) throws ENOTDIR. This is the failure
#    mode test 4 in mcp-metrics-snapshot-auto-persist.test.js (Phase 102).
file .planning/metrics 2>/dev/null

# 5. Direct invoke to isolate the failure — call persistSnapshot() outside
#    the MCP handler. Same code path, no MCP transport noise.
node -e "import('./src/core/metrics.js').then(m => m.persistSnapshot().then(p => console.log('OK', p)).catch(e => console.error('FAIL', e.code, e.message)))"
```

**Fix**

| Diagnosis | Fix |
|---|---|
| `ENOSPC` (disk full) | Free disk space. Older snapshots are auto-cleaned at >30d (Phase 99 retention), but in-flight saturation can still trip. After freeing space, the next tool call writes a fresh snapshot. |
| `EACCES` / permission denied | `chmod -R u+rwX .planning/metrics/` (or `icacls` on Windows). The directory was likely created with stricter umask by another process or root. |
| `.planning/metrics` exists as a regular file (ENOTDIR) | `rm .planning/metrics` then re-trigger any tool call. The auto-snapshot path will recreate the directory on demand. |
| Permission denied via security software (Windows Defender, antivirus quarantine on `.json` writes) | Whitelist `.planning/metrics/snapshots/` (or the project root) in the AV exclusion list. Same fix shape as Scenario 5 antivirus quarantine. |
| `EROFS` (read-only mount) | The project is on a read-only mount (e.g. squashfs, NFS export with `ro`). Move the project to a writable location, or set `KIT_MCP_METRICS_DIR=<writable-path>` if/when that env override ships (currently absent — file an issue). |
| Throttle skips (no error, just empty) | Expected — the in-memory `_lastAutoPersistTs` guard suppresses writes within 1s of each other (Phase 102 throttle decision). Check that you ran two tool calls > 1s apart before declaring failure. |

**Verification**

```bash
# 1. stderr clean after the fix — call the tool again and watch stderr.
# 2. New snapshot file appears in the directory:
ls .planning/metrics/snapshots/*.json 2>/dev/null | wc -l
# Expected: at least one fresh file with mtime in the last few seconds.

# 3. /burn-rate-status command no longer returns no_data once at least 1
#    snapshot covers the lookahead window (1h fast / 6h slow per Phase 103).
```

> **Cross-ref:** Phase 102 (auto-snapshot graceful design) at [`.planning/phases/102-auto-snapshot-metrics-tool/`](./phases/102-auto-snapshot-metrics-tool/) and Phase 99 (retention + burn-rate calc) at [`.planning/phases/99-metrics-retention-burnrate-calc/`](./phases/99-metrics-retention-burnrate-calc/). The stderr message format is fixed in [`src/mcp-server/index.js`](../src/mcp-server/index.js) handleMetricsSnapshot wrap.

## 8. Multi-IDE sidecar port collision

**Symptom**

- The user opens the same project in Claude Code AND Cursor (or any 2+ IDEs) at the same time.
- Both auto-spawn a sidecar (default base port 7100) and the second one escalates to 7101..7199 per [`src/ui/port.js`](../src/ui/port.js) auto-port logic.
- Hooks fire from IDE A, but the UI sees them in IDE B (or vice versa) — events route to the wrong sidecar because the project's shared `.kit-mcp/sidecar.lock` records only one PID/port pair at a time.

**Diagnosis**

```bash
# 1. List sidecars currently bound to the 71xx range.
ss -tlnp 2>/dev/null | grep -E '710[0-9]|71[0-9]{2}' || netstat -an | grep 71

# 2. Inspect the lockfile — pid + port + startedAt + ttlSec are recorded.
#    A shared lockfile means both IDEs see the same record; only the most
#    recent writer wins.
cat .claude/.kit-mcp/sidecar.lock 2>/dev/null || cat .kit-mcp/sidecar.lock 2>/dev/null

# 3. Hit each suspected port's healthz to differentiate active sidecars.
curl -fsS http://localhost:7100/healthz
curl -fsS http://localhost:7101/healthz

# 4. If you cannot tell which IDE owns which sidecar, kill them and observe
#    which IDE re-spawns first.
ps -ef | grep -E 'kit-mcp|sidecar' | grep -v grep
```

**Fix**

| Diagnosis | Fix |
|---|---|
| Both IDEs share the same project lockfile, last writer wins | Per-IDE port base override: in IDE B's environment (settings → MCP server env, or shell rc when launching that IDE), set `KIT_MCP_UI_PORT_BASE=7200`. The two sidecars then bind to disjoint ranges (7100-7199 and 7200-7299) and stop fighting for the lockfile. |
| Stale lockfile from IDE A blocks IDE B's spawn | `kit ui stop --project-root .` is idempotent. Then IDE B's next tool call will auto-spawn cleanly into the cleared lockfile slot. Same recovery shape as Scenario 2 stale-lockfile. |
| You want one shared sidecar (single UI surface for both IDEs) | Set `KIT_MCP_UI_PORT=7100` (note: `_PORT`, not `_PORT_BASE`) in BOTH IDEs. The first one to spawn wins; the second one connects as client and emits hooks into the shared event bus. Both IDEs see the same events. |
| Hooks route to wrong sidecar in spite of disjoint ranges | Check the hook publisher's `KIT_MCP_UI_PORT` env. The publisher uses the lockfile by default; setting `KIT_MCP_UI_PORT` explicit pins the hook to a specific sidecar regardless of which one wrote the lockfile last. See [`src/ui/auto-spawn.js`](../src/ui/auto-spawn.js) and the Phase 21 hook publisher. |
| Both IDEs run different kit-mcp versions (one stale via `npx -y` cache) | Re-run `npx -y @luanpdd/kit-mcp@latest --version` in each IDE shell. The IDE caches `npx` resolutions; a stale version can spawn a different sidecar build entirely. Restart the IDE after upgrading. |

**Verification**

```bash
# Each IDE should report its own active sidecar with a distinct port:
# (run inside each IDE's environment)
kit ui status --project-root .                            # → "🟢 running on http://localhost:71xx"
# Different x's per IDE; no port conflict reported.
```

## 9. Critical CVE blocks publish

**Symptom**

- `git push origin v<X.Y.Z>` succeeded, but the GitHub Actions release workflow shows red on the audit step with `npm audit found 1 high severity vulnerability`.
- A direct dep (`@modelcontextprotocol/sdk`, `commander`, `picocolors`) or a transitive dep is flagged.
- The CI gate `Audit — npm audit (high/critical CVEs in runtime deps)` exits non-zero — see the rule in [`ci.yml`](../.github/workflows/ci.yml) which runs `npm audit --omit=dev --audit-level=high` against `dependencies` and `optionalDependencies` only.

**Diagnosis**

```bash
# 1. Inspect the failing run — the audit output names the package + CVE.
gh run list --workflow release.yml --limit 5
gh run view <run-id> --log-failed | head -50

# 2. Reproduce the audit gate locally with the same flags as CI.
npm audit --omit=dev --audit-level=high

# 3. If the flagged package is transitive, surface the chain to figure out
#    which direct dep brought it in.
npm ls <flagged-package>

# 4. Read the CVE description — DoS-only theoretical CVEs are different
#    from RCE / data exfiltration / auth bypass severity.
# Visit https://github.com/advisories/GHSA-<id> from the audit output.
```

**Fix**

| Diagnosis | Fix |
|---|---|
| Direct dep has a patched version available | `npm install <dep>@<patched-version>`, commit, retag (`git tag -d v<X.Y.Z> && git push --delete origin v<X.Y.Z> && git tag v<X.Y.Z> && git push --tags`). |
| Transitive dep has the vulnerability, direct dep won't ship a fix yet | Use [`npm overrides`](https://docs.npmjs.com/cli/v10/configuring-npm/package-json#overrides) in `package.json` to pin the transitive to a patched version. Commit, retag. Document the override + CVE link in `package.json` comments (line above the `overrides` block) for future maintainers. |
| Direct dep is unmaintained / no patch in sight | File an issue against the direct dep maintainer. If the CVE is theoretical-only DoS (no RCE, no data leak) and you must ship, **temporarily** flip the audit gate from `--audit-level=high` to `--audit-level=critical` in `ci.yml` with an inline comment (`# Temporary: <CVE-id> tracked at #<issue>; revert when patched`) and an open issue blocking the next release. Never bypass without an issue and a revert plan. |
| `NPM_TOKEN` problem masquerades as audit fail (red workflow, vague error) | Re-read the workflow log carefully. Token errors usually say `403` / `401`; audit errors say `found N high vulnerabilities`. They look superficially similar in summary view. |
| New direct dep added in the same PR pushed total > 6 budget | Different gate — `Audit — runtime deps budget (REQ INFRA-17-01)` fails first. Drop the new dep or follow the budget-bump protocol (ADR in `.planning/decisions.md` + budget number bump in `ci.yml`). |

**Verification**

```bash
# 1. Local audit gate exits 0:
npm audit --omit=dev --audit-level=high
echo "audit exit: $?"

# 2. Retag and re-fire the workflow:
git tag -d v<X.Y.Z>
git push --delete origin v<X.Y.Z>
git tag v<X.Y.Z>
git push --tags

# 3. Workflow turns green; npm registry shows the new version.
gh run list --workflow release.yml --limit 1                 # status: success
npm view @luanpdd/kit-mcp version                            # equals tagged version
```

> **Cross-ref:** Phase 92.01 (audit gate at high severity) at [`.planning/phases/92-deps-budget-runtime/`](./phases/92-deps-budget-runtime/), Phase 89 (manifest regen at `prepublish`) at [`.planning/phases/89-cold-start-perf/`](./phases/89-cold-start-perf/), and the [`hermetic-builds`](../kit/skills/hermetic-builds/SKILL.md) skill — its "audit gate is the only defense, intentionally" rationale is the canonical reason this RUNBOOK section exists.

## SLO check — is the service degraded?

When unsure whether a user-reported issue is "broken" vs "slow", check the SLOs:

```bash
# 1. Drive a metrics-snapshot via your MCP client OR via the integration test pattern.
#    The simplest in-repo way:
node test/integration/mcp-metrics-snapshot.test.js | tail -20

# 2. Compute SLI from the snapshot:
#    Availability = sum(*:ok) / (sum(*:ok) + sum(*:error))
#    Latency      = max over tools of latency.<tool>.p95

# 3. Compare against targets in:
cat .planning/slos/mcp-tool-availability.yml | grep target
cat .planning/slos/mcp-tool-latency.yml | grep target_ms
```

| SLO | Target | Source |
|---|---|---|
| MCP tool availability | 99.5% (30d sliding) | [`mcp-tool-availability.yml`](./slos/mcp-tool-availability.yml) |
| MCP tool latency p95 | ≤ 200 ms (30d sliding) | [`mcp-tool-latency.yml`](./slos/mcp-tool-latency.yml) |

If SLI < target → the service is officially degraded; declare an incident, follow the [`blameless-postmortems`](../kit/skills/blameless-postmortems/SKILL.md) skill template after recovery. If SLI ≥ target but the user is still unhappy → the issue is below the SLO threshold (often a UX or doc gap, not a service degradation).

> **Cross-ref:** the `/burn-rate-status` command (shipped in v1.10) reads these YAML files plus a recent snapshot and renders a burn-rate table. See [`.claude/commands/burn-rate-status.md`](../.claude/commands/burn-rate-status.md).

## Escalation paths

There is no on-call rotation today (single-repo, single-human maintainer). Escalation is:

1. **First** — open a GitHub issue at [`luanpdd/kit-mcp/issues`](https://github.com/luanpdd/kit-mcp/issues) with `node bin/cli.js doctor --json` output attached.
2. **Security** — use [`SECURITY.md`](../SECURITY.md) (private disclosure) for anything that looks like an attack signal (manifest tamper, RCE attempt, path traversal). Do **not** publish exploit details to a public issue.
3. **Discord** — for community help on ambiguous cases, the framework Discord is linked from [`/entrar-discord`](../.claude/commands/entrar-discord.md).
