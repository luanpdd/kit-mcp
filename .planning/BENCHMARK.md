# BENCHMARK — kit-mcp baseline performance envelope

> Reference numbers captured on a single developer machine, used as the "before" line for detecting regressions in future phases. Companion to [`FAILURE-MODES.md`](./FAILURE-MODES.md) (modes 7 and 8 are the perf-regression budget) and to the SLOs at [`./slos/`](./slos/) (the latency SLO derives its 200ms target from these numbers).
>
> **Not** a load test or production capacity model. kit-mcp is a developer-tool MCP server launched on demand by an IDE; "throughput" is bounded by the LLM agent's request rate, not by the server. The numbers here describe the *user-perceived latency floor* on a single dispatch.
>
> Generated by Phase 96 (v1.18 — Eat Your Own Dog Food). M4 last refreshed 2026-05-10 (Phase 105 / SRE-20-02 — pre-warm kit cache).

## Baseline metrics

### M1 — Cold start of `kit list-agents --terse`

The end-to-end first-byte latency for the simplest read-only CLI command, exercising kit load + table render. Each invocation spawns a fresh `node` process (no cache reuse).

| Run | Wall time |
|---|---|
| 1 | 245.6 ms |
| 2 | 232.4 ms |
| 3 | 222.0 ms |
| 4 | 232.6 ms |
| 5 | 218.9 ms |
| **Median** | **232.4 ms** |
| min / max | 218.9 / 245.6 ms |

**How to reproduce**

```bash
for i in 1 2 3 4 5; do
  node -e "const start = process.hrtime.bigint();
           require('child_process').execSync('node bin/cli.js kit list-agents --terse', {stdio:'pipe'});
           console.log((Number(process.hrtime.bigint() - start)/1e6).toFixed(1));"
done
```

**Why this number matters.** This is the floor for *any* CLI command — every other command spawns the same Node + does at least the same kit load. Regression here regresses everything. The Phase 89.01 lazy-import work and the Phase 90.01 manifest cache both target this path; a future phase that adds eager imports will show up as an N-ms cliff in this row.

### M2 — Sync wall time (`kit sync install claude-code`, 321 files)

Full sync of the bundled kit into a clean target directory, then two re-runs to measure the diff-filter steady state.

| Run | Wall time | Notes |
|---|---|---|
| 1st (cold — empty target, all writes) | 503 ms | full tree copy + reference renders |
| 2nd (warm — diff filter active) | 404 ms | Phase 91.01 mtime+size heuristic skips re-write |
| 3rd (steady-state) | 391 ms | floor for the filtered path |

**How to reproduce**

```bash
mkdir -p /tmp/sync-bench && cd /tmp/sync-bench
rm -rf .claude

for label in "1st-cold" "2nd-warm" "3rd-steady"; do
  start=$(date +%s%N)
  node "$REPO_ROOT/bin/cli.js" sync install claude-code --project-root /tmp/sync-bench >/dev/null
  end=$(date +%s%N)
  echo "$label: $(( (end - start) / 1000000 )) ms"
done
```

**Why this number matters.** The Phase 91 decision diaries called out a target of "warm sync ≤ 1s" — the steady-state 391ms is well under that. Regression budget: **2× cold (≤1000ms) before treating as a failure mode**, per [FAILURE-MODES.md](./FAILURE-MODES.md) mode 7. Personal kits (custom `kit_root`) with 1000+ files are the natural regression boundary; if a user reports >2s, it's time to add a perf gate test.

### M3 — Memory footprint after kit load

Resident set + V8 heap *after* a full `listKit` of the bundled kit (47+87+45 entries) settled.

| Field | Value |
|---|---|
| RSS | 53.0 MB |
| Heap (used) | 8.5 MB |
| Heap (total) | 14.3 MB |
| External | 2.2 MB |
| ArrayBuffers | 0.7 MB |

**How to reproduce**

```bash
node -e "
import('./src/core/kit.js').then(async (m) => {
  const k = await m.listKit(m.BUNDLED_KIT_ROOT);
  await new Promise(r => setTimeout(r, 50));
  const u = process.memoryUsage();
  console.log({
    rssMB:        +(u.rss/1024/1024).toFixed(1),
    heapUsedMB:   +(u.heapUsed/1024/1024).toFixed(1),
    heapTotalMB:  +(u.heapTotal/1024/1024).toFixed(1),
    externalMB:   +(u.external/1024/1024).toFixed(1),
    arrayBuffersMB:+(u.arrayBuffers/1024/1024).toFixed(1),
    counts: { agents: k.agents.length, commands: k.commands.length, skills: k.skills.length },
  });
});
"
```

**Why this number matters.** kit-mcp runs as an IDE child process; an MCP server that grows a 200MB heap on idle is a friction point for the host. The 53 MB RSS / 14 MB V8 heap profile is small enough that the LLM agent itself dominates the IDE's memory budget by orders of magnitude. Regression budget: **2× current (≤100 MB RSS)** before re-investigating; Phase 92.01 budget-of-3-deps decisions explicitly trade off RSS for dep count.

### M4 — MCP tool dispatch latency (in-server, p50/p95/p99)

End-to-end MCP-tool latency as observed by the Phase 94.01 metrics module — wall-clock time from dispatcher entry to handler return, captured by `recordLatency` in [`src/core/metrics.js`](../src/core/metrics.js). The SLI consumed by [`mcp-tool-latency.yml`](./slos/mcp-tool-latency.yml).

#### v1.20.0 — pre-warm kit cache (Phase 105 / SRE-20-02)

Probe: 30 `kit list-agents --terse` dispatches against a fresh server (boot-time `KIT_MCP_METRICS_RESET=1`), with an 800ms post-initialize wait so the boot-time pre-warm (Phase 105 — fire-and-forget `listKit(BUNDLED_KIT_ROOT)` after `server.connect`) has resolved before the first dispatch. Five runs, headline = median; range in parens.

| Tool | Count | p50 | p95 | p99 |
|---|---|---|---|---|
| `kit` | 30 | **0.0 ms** | **0.0 ms** *(0–0.55 ms)* | **0.0 ms** *(0–1.0 ms)* |

**Reading the numbers.** Phase 105 added a fire-and-forget `listKit(BUNDLED_KIT_ROOT).catch(() => {})` immediately after `server.connect(transport)` in `startStdio()`. This shifts the ~144ms cold-path disk read (47 agents + 87 commands + 45 skills + frontmatter parse on ~180 files) from the first user-visible request to the boot path, where it is invisible behind IDE startup. Real LLM clients take seconds-to-minutes to issue the first `tools/call` after IDE startup; pre-warm completes in ~140ms, so by the time the first dispatch lands the cache is warm. p50/p95/p99 collapse to ~0ms because every dispatch is now a cache hit — there is no cold path left to dominate the tail.

**Δ vs v1.17.0:** p95 144.55 ms → 0.0 ms (~100% reduction). Far exceeds the ROADMAP target of "≥30% redução vs 144ms baseline" (and exceeds the sub-100ms target). Regression budget should now treat any p95 > 50ms as a likely pre-warm regression; previous SLO target of 200ms p95 (`./slos/mcp-tool-latency.yml`) remains conservatively wide and still serves as the outer safety net.

**Trade-off note.** Pre-warm is fire-and-forget — `server.connect` returns immediately, no boot delay. The race window is the rare case where a client issues `tools/call` against `kit` within ~140ms of `server.connect` returning; the dispatch then races the pre-warm. Both paths populate the same `kitCache`, so the cost is ~140ms either way (no double work, no incorrectness). No fallback to blocking await was needed because the steady-state win is already complete.

**How to reproduce**

Drive a fresh MCP server with the post-initialize wait (so pre-warm completes before the first dispatch):

```bash
node .planning/phases/105-prr-performance-5-5/probe-final.mjs  # 5 runs, prints summary
```

Or use the canonical integration probe (which uses 200ms perStep — also enough for pre-warm to complete on most hardware):

```bash
node test/integration/mcp-metrics-snapshot.test.js  # exits 0 on green
```

Or, for an ad-hoc probe over N=30, see the test scaffold itself ([`test/integration/mcp-metrics-snapshot.test.js`](../test/integration/mcp-metrics-snapshot.test.js)) — replicate the `driveServer` helper with 30 `tools/call` requests and read the `metrics-snapshot` envelope.

#### v1.17.0 — pre-warm absent (baseline) [archived]

Probe: 30 `kit list-agents --terse` dispatches against a fresh server (boot-time `KIT_MCP_METRICS_RESET=1`), then `metrics-snapshot` to read the histogram.

| Tool | Count | p50 | p95 | p99 |
|---|---|---|---|---|
| `kit` | 30 | 0.0 ms | 144.55 ms | 146.42 ms |

**Reading the numbers (v1.17.0).** p50 is 0 because the kit cache (`src/core/kit.js` `kitCache`) is hot after the first dispatch — subsequent calls return cached structures. The p95/p99 fall on the *cold* dispatches (first 1–2 calls do the disk read). This shape is exactly what the Phase 94.01 design predicted: bounded histogram, hot path in microseconds, cold path under 200ms. **Phase 105 closed this gap by moving the cold path to boot.**

**Why this number matters.** The latency SLO target is **p95 ≤ 200ms** ([`./slos/mcp-tool-latency.yml`](./slos/mcp-tool-latency.yml)). v1.17.0 p95 of 144.55ms gave ~55ms headroom; v1.20.0 widens that to the entire 200ms budget because p95 is now ~0ms. Regression that pushes p95 above 200ms trips the SLO and starts burning the error budget; consult [`burn-rate-status`](../.claude/commands/burn-rate-status.md). Cold-path improvements (e.g. moving the Phase 90 manifest cache earlier in boot) would shrink p95; eager imports on dispatch would inflate it.

### M5 — Tarball size (`npm pack --dry-run`)

The published artifact's footprint, gating user `npm install` time and disk impact.

| Field | Value |
|---|---|
| Package size | 1.1 MB |
| Unpacked size | 3.6 MB |
| Total files | 384 |
| Tarball name | `luanpdd-kit-mcp-1.17.0.tgz` |

**How to reproduce**

```bash
npm pack --dry-run 2>&1 | tail -10
```

**Why this number matters.** The tarball includes the entire bundled `kit/` (the 321-file fixture used by sync demos and tests) plus `src/`, `bin/`, `test/`, framework templates. The 1.1 MB / 384 file profile is small enough to install over a coffee-shop wifi. Regression budget: **2× current (≤2.2 MB packed)** before investigating; the natural growth driver is users adding their personal kit content into `kit/` — if that happens, the long-term answer is `kit_root` env var (already supported), not bloating the tarball.

## Refresh policy

- Re-run **after every milestone** that touches perf-relevant code paths — sync, kit loader, lazy imports, manifest verify, MCP dispatcher.
- Update both the per-metric tables and the summary above; do **not** delete the previous row — version it inline (`# v1.17.0` / `# v1.18.0`) so the trend is visible at a glance.
- Single-point measurements only. We don't aggregate across machines because milliseconds are too noisy at this scale; the *trend on the same kind of machine* is the signal, not absolute portability.
- If a future CI gate captures cold start automatically (candidate work — see [FAILURE-MODES.md](./FAILURE-MODES.md) mode 8 follow-up), this doc becomes the historical record and the gate becomes the live tripwire.
