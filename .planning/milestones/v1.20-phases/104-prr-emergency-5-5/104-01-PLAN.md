**Scenario 6: CI coverage gate regression**
- Symptom: PR/push fails CI line-coverage gate `::error::line coverage XX% < 86% threshold` after Phase 100 ratchet (v1.20).
- Diagnosis: 1) `node --experimental-test-coverage --test test/unit/*.test.js > /tmp/cov.txt`; 2) `grep "^ℹ all files" /tmp/cov.txt` → see actual %; 3) `grep "^ℹ src/" /tmp/cov.txt | sort -t'|' -k2 -n` → identify regressed files
- Fix table: regressed file due to new uncovered lines (add tests targeted), removed test by mistake (`git log -p test/unit/`), threshold too aggressive (rollback ci.yml THRESHOLD with rationale + open issue), v1.21+ ratchet path documented
- Verification: re-run coverage locally; gate passes; PR turns green
- Cross-ref Phase 100 + ci.yml THRESHOLD history block

**Scenario 7: Auto-snapshot persist failure**
- Symptom: stderr stream shows `[kit-mcp] auto-snapshot persist failed: <error>` repeatedly. Handler still returns payload (per Phase 102 graceful design), but `.planning/metrics/snapshots/` is empty/stale; `/burn-rate-status` reports `no_data`.
- Diagnosis: 1) Check error type from stderr (ENOSPC, EACCES, EISDIR, ENOTDIR, EROFS); 2) `df -h .planning` (disk full?); 3) `ls -la .planning/metrics/snapshots/` (writable?); 4) `node -e "import('./src/core/metrics.js').then(m => m.persistSnapshot())"` direct invoke to isolate the failure
- Fix table: ENOSPC (free space), EACCES (chmod or chown), `.planning/metrics` exists as file not dir (rm + retry), permission denied via security software (whitelist)
- Verification: stderr clean after fix, `ls .planning/metrics/snapshots/*.json | wc -l` shows new files appearing on next tool call, `/burn-rate-status` returns data
- Cross-ref Phase 102 + Phase 99

**Scenario 8: Multi-IDE sidecar port collision**
- Symptom: User opens project in Claude Code AND Cursor (or any 2+ IDEs). Both auto-spawn sidecar. Hooks fire in IDE A, but UI in IDE B shows nothing or vice-versa.
- Diagnosis: 1) `ss -tlnp 2>/dev/null | grep -E '710[0-9]|71[0-9]{2}'` (or `netstat -an | grep 71` Windows); 2) Inspect each IDE's lockfile if separate `.kit-mcp/` per project — look at PID + port fields; 3) `curl http://localhost:7100/healthz && curl http://localhost:7101/healthz` differentiate active sidecars
- Fix table: shared `.kit-mcp/` (set per-IDE `KIT_MCP_UI_PORT_BASE=7200` in IDE B environment), kill stale sidecar from IDE A first, accept duplicate sidecars (one per IDE) and route hooks via `KIT_MCP_UI_PORT` env to specific sidecar
- Verification: each IDE shows independent sidecar UI; hooks route correctly; no port conflicts
- Cross-ref Phase 13/14 (sidecar architecture v1.2) + Phase 21 (hook publisher)

**Scenario 9: Critical CVE blocks publish**
- Symptom: `git push origin v<X.Y.Z>` succeeds, but release.yml workflow red with `npm audit found 1 high severity vulnerability`. `@modelcontextprotocol/sdk` or `commander` or transitive dep flagged.
- Diagnosis: 1) `gh run view <run-id> --log-failed | head -50`; 2) `npm audit --omit=dev --audit-level=high` locally (parity with CI gate); 3) `npm ls <flagged-package>` shows transitive chain; 4) Check CVE description (data-only? RCE? bypass?)
- Fix table: pin direct dep version (forces resolution avoiding flagged transitive), upgrade direct dep to version with patched transitive, `npm overrides` clause in package.json for stuck transitives, file issue against direct dep maintainer + temporary `--audit-level critical` rollback if CVE is theoretical-only DoS
- Verification: `npm audit --omit=dev --audit-level=high` exits 0; retag and re-run release workflow turns green; npm registry shows new version
- Cross-ref Phase 92.01 (audit gate) + Phase 89 (manifest) + hermetic-builds skill
