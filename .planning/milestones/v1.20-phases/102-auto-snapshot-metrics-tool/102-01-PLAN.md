# Plano 102-01: Auto-snapshot em metrics-snapshot MCP Tool

## Objetivo

Tool MCP `metrics-snapshot` automaticamente persiste snapshot via `metrics.persistSnapshot()` em vez de exigir trigger manual externo. Idempotência via in-memory `lastPersistTs` guard (1s throttle).

## Tasks

<read_first>

- src/core/metrics.js (linhas 160-225 para signatures de persistSnapshot/loadSnapshots/snapshot)
</read_first>

<action>
1. Em `src/mcp-server/index.js`, no topo do arquivo (logo após imports), adicionar import de `persistSnapshot`:
   ```js
   import { snapshot as metricsSnapshot, persistSnapshot } from '../core/metrics.js';
   ```
   (verificar se já existe import — pode ser que apenas `metricsSnapshot` esteja importado).

2. Adicionar antes da declaração de `handleMetricsSnapshot` (~linha 308):
   ```js
   // OBS-20-01 (Phase 102): auto-persist throttle — clients polling rapidly
   // shouldn't create N files per second. 1s is generous vs typical 30s+ polls.
   // State is in-memory; resets on server restart.
   let _lastAutoPersistTs = 0;
   const AUTO_PERSIST_THROTTLE_MS = 1000;
   ```

3. Substituir handler atual `async function handleMetricsSnapshot()` por:
   ```js
   async function handleMetricsSnapshot() {
     const payload = metricsSnapshot();
     const now = Date.now();
     if (now - _lastAutoPersistTs >= AUTO_PERSIST_THROTTLE_MS) {
       try {
         await persistSnapshot();
         _lastAutoPersistTs = now;
       } catch (err) {
         // OBS-20-01: graceful — log para stderr, não falha o handler.
         // Snapshot in-memory ainda retornado normalmente.
         process.stderr.write(`[kit-mcp] auto-snapshot persist failed: ${err.message}\n`);
       }
     }
     return payload;
   }
   ```
</action>
