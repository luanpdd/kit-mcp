# Plano 101-01: Setup Stryker + Baseline Mutation Score em src/core/

**Objetivo:** Adicionar `@stryker-mutator/core` como dev dep, configurar `stryker.config.json` para `src/core/`, criar script `npm run test:mutation`, atualizar `.gitignore`, executar baseline e documentar resultado em `.planning/audits/v1.20/MUTATION-BASELINE.md`. Zero mudanças funcionais; gate v1.21+ depende deste baseline.

## Contexto

REQ INFRA-20-02 fecha o item parqueado pós-v1.19 "mutation testing (stryker)". Phase 100 entregou suite robusta (651 testes, 86.84% coverage) — base sólida para medir mutation score real. Esta fase é instrumentação canônica: única fase do milestone que adiciona dep nova (devDep apenas, budget runtime intacto). Não bloqueia CI nesta versão.

Stryker é o mutation tester padrão de facto para Node.js/JS. Integração com `node:test` (test runner nativo) usa `command` runner em vez de plugin dedicado, com trade-off de `coverageAnalysis: 'off'` (sem perTest hooks). Mutator scope inicial: 15 arquivos em `src/core/` (registry, kit, sync, reverse-sync, gates, gate-runner, metrics, error-redaction, manifest-verify, path-safety, ui, reflect, watch, replays, failures).

<title>Criar stryker.config.json scoped to src/core/ + node:test command runner</title>
<description>

```json
{
  "$schema": "./node_modules/@stryker-mutator/core/schema/stryker-schema.json",
  "_comment": "kit-mcp Phase 101 — baseline mutation testing scoped to src/core/. CI execution is opt-in (npm run test:mutation). Reports to reports/mutation/ (gitignored).",
  "packageManager": "npm",
  "mutate": [
    "src/core/**/*.js",
    "!src/core/**/*.test.js"
  ],
  "testRunner": "command",
  "commandRunner": {
    "command": "node test/run.mjs test/unit"
  },
  "coverageAnalysis": "off",
  "reporters": [
    "html",
    "json",
    "clear-text"
  ],
  "htmlReporter": {
    "fileName": "reports/mutation/mutation-report.html"
  },
  "jsonReporter": {
    "fileName": "reports/mutation/mutation-report.json"
  },
  "tempDirName": ".stryker-tmp",
  "timeoutMS": 60000,
  "concurrency": 4,
  "logLevel": "info"
}
```

Notas:
- `mutate` glob: `src/core/**/*.js` cobre os 15 arquivos do core (registry, kit, sync, etc.) — exclui qualquer `*.test.js` se algum existir.
- `testRunner: "command"` é compatível com `node:test` (não há plugin dedicado oficial). `commandRunner.command` deve replicar EXATAMENTE o entrypoint usado em `npm test` para evitar drift.
- `coverageAnalysis: "off"` é necessário porque `node:test` não suporta perTest hooks. Trade-off: cada mutant roda toda a suite — execução mais lenta, porém compatível.
- `concurrency: 4` é conservador para evitar sobrecarga em laptop dev. Pode ser ajustado em fases futuras.
- `timeoutMS: 60000` (1 minuto por mutant) — generoso para suite atual (~5-10s por full run).
- Reports paths sob `reports/mutation/` — único diretório criado por Stryker; gitignored em Tarefa 3.
</description>

<task id="5" parallel-group="baseline-run" estimated-minutes="25" depends-on="4">
<title>Rodar baseline `npm run test:mutation` em src/core/ e capturar JSON report</title>
<description>
Executa Stryker baseline em src/core/ (15 arquivos). Tempo esperado: 15-30 min dependendo do hardware (cada mutant ~5-10s × ~500-1000 mutants estimados).

Steps:
1. Run `npm run test:mutation` na raiz. Stryker irá:
   - Detectar mutators aplicáveis (arithmetic, conditional, string-literal, etc.)
   - Para cada mutant: aplicar transformação no AST de um arquivo em src/core/, rodar `node test/run.mjs test/unit`, classificar como `killed` (teste falhou — bom), `survived` (teste passou — gap de coverage comportamental), `timeout` (mutant infinite loop — count as killed por convenção), ou `noCoverage` (linha não exercida — count como negative signal).
2. Aguardar conclusão. Stryker imprime progress periódico (clear-text reporter ativo). Não interromper.
3. Após terminar:
   - `reports/mutation/mutation-report.html` — relatório navegável (abrir manualmente para inspeção)
   - `reports/mutation/mutation-report.json` — JSON estruturado para parsing automatizado
4. Capturar score final do clear-text output (linha tipo `Mutation score: XX.XX%`).
5. Salvar uma cópia do summary stdout em `.planning/audits/v1.20/STRYKER-RUN-LOG.txt` para referência (gitignore-exempt — pode comitar para auditoria, OU adicionar ao gitignore se preferir; default: comitar).

Se `npm run test:mutation` falhar (não confunde com mutation score baixo — falha real é exit code != 0 antes de qualquer mutant):
- Erro de config: revisar stryker.config.json (especialmente `commandRunner.command` e mutate glob).
- Erro de network: Stryker não requer network após instalação; se houver erro de proxy, configurar `npm config`.
- Erro de timeout no setup: aumentar `timeoutMS` em config.

Se score for muito baixo (<40%): documentar honesto na BASELINE.md — não há "fail" técnico, é só sinal de que comportamental coverage tem gaps. Esse é o ponto da fase.
</description>
