Alternativa: branch coverage como 2º gate (atualmente 83.58%).
- THRESHOLD 86 → 90 ratchet — depende de cli/index.js extract para ser honesto sem violar Stable API v1.0+. 3 avenues canônicas documentadas inline em ci.yml.

**Phase 101 (Mutation):**
- Baseline 10/15 files complete — restantes (`sync.js`, `ui.js`, `watch.js`, `reverse-sync.js`, `gate-runner.js`) documentados na Avenue A do MUTATION-BASELINE.md. Sessão dedicada para fechar.
- Mutation gate em CI threshold ~55% — depende de baseline 100% complete + decisão sobre cache/parallelism (run paralelo, cache `.stryker-tmp/`, run apenas em PRs que tocam `src/core/**`).
- Critical files push: `registry.js` 18.62% → ≥60% (TARGETS table behavior assertions); `path-safety.js` 17.24% → ≥80% (assert exact rejection messages); `replays.js` 45.36% → ≥70% (property-based tests com fast-check).

**Phase 105 (Performance):**
- p99 latency monitoring sustained sob load real — atualmente é p95 only via metrics-snapshot in-memory; v1.21+ pode adicionar disk-persistent snapshots para historical trend.
- CI gate auto-probe M4 + fail on regression > 50ms p95 — atual refresh policy é manual ("re-run após cada milestone que toca paths perf-relevantes"). Auto-CI gate caçaria regressões antes do merge.
- Cold-start CLI sub-200ms (M1) — atual 232ms median; requer lazy-load mais agressivo de `commander`. Diminishing returns vs Phase 16/89.

**Drill cadence (Phase 104):**

- Auto-trigger drill via cron (monthly) — overkill para single-human cadence atual.
