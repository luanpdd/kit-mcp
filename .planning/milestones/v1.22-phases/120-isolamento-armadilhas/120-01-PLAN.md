## REQs cobertos

### ISOLAMENTO (skill 1)

- ✅ **ISOLAMENTO-01**: 6 race conditions documentados com SQL exemplo isolado de cada — dirty read (Postgres NÃO permite, READ UNCOMMITTED = READ COMMITTED), dirty write (prevenido por row-level lock), read skew (prevenido por REPEATABLE READ via MVCC), lost update (read-modify-write concorrente), write skew (predicate decision invalidation com exemplo doctor on-call), phantom read (WHERE + INSERT cross-trx). Termos EN preservados.
- ✅ **ISOLAMENTO-02**: Árvore de decisão isolation level — READ COMMITTED (default 95% CRUD simples), REPEATABLE READ (snapshot consistent para queries longas multi-table), SERIALIZABLE (SSI predicate-aware para invariantes complexos cross-row).
- ✅ **ISOLAMENTO-03**: 3 padrões para prevenir lost update tabulados com tradeoff — FOR UPDATE (pessimista, baixo throughput se contention), CAS otimista via `WHERE version = $v` (alto throughput, retry on miss), `pg_advisory_xact_lock(hashtext('name'))` (semântico, lock global singleton).
- ✅ **ISOLAMENTO-04**: Prevenção write skew em 3 caminhos — FOR UPDATE em todas rows lidas no predicate (materializa conflito), `EXCLUDE USING gist` constraint para overlap (room booking), `SET TRANSACTION ISOLATION LEVEL SERIALIZABLE` (Postgres SSI predicate-aware) como fallback genérico.
- ✅ **ISOLAMENTO-05**: Prevenção phantom read via SSI predicate-aware desde Postgres 9.1; contraste explícito com REPEATABLE READ que NÃO previne phantom (apenas snapshot da sessão, não cobre INSERT cross-trx que cai no predicate).

### ARMADILHAS (skill 2)

- ✅ **ARMADILHAS-01**: Tabela canônica de timestamps Postgres 4 funções (`now()` / `transaction_timestamp()` / `statement_timestamp()` / `clock_timestamp()`) com semântica + quando usar / quando NÃO. Regra absoluta: NUNCA `clock_timestamp()` para expiração de TTL/lease/ordenação cross-transaction.
- ✅ **ARMADILHAS-02**: Fencing tokens canônicos via `pg_advisory_xact_lock(hashtext('lock_name'))` + `CREATE SEQUENCE fencing_tokens_seq` monotônico + storage rejeita writes com `last_token < $token`. Aplicações tabuladas: super-admin TTL impersonation, jobs pgmq agendados, eleição de líder.
- ✅ **ARMADILHAS-03**: Cenário canônico GC pause / process pause — Nó A adquire lease 30s, Nó A sofre GC pause 45s, lease expira, Nó B assume liderança, Nó A volta achando que tem lease → split brain (escrita dupla). Mitigação: fencing token monotônico crescente; storage rejeita writes com token < último visto.
- ✅ **ARMADILHAS-04**: Falhas parciais — detecção por timeout é falaciosa (lentidão ≠ morte). Pattern phi accrual failure detector (variance de heartbeats, probabilístico). Regra prática Supabase: assumir nó morto apenas após `timeout >= 3× P99 RTT histórico` + consenso de N-1 nós.
- ✅ **ARMADILHAS-05**: Modelos de sistema mapeados — crash-stop (irreal), crash-recovery (**modelo Supabase típico** — Edge Function pode reiniciar, Postgres pode failover), byzantine (fora do scope, apenas blockchain/safety-critical). Tabela canônica com aplicação em Supabase.
