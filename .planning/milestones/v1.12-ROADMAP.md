## Visão geral do milestone

Adicionar 5ª suíte ao kit (Legacy) — derivada do livro Feathers e modernizada para 2026. v1.12 é **content-only por design** — zero alterações em `src/core/`. Stable API v1.0+ preservada. Conteúdo PT-BR alinhado.

**Princípio editorial:** cada artefato marca explicitamente o que é "Feathers original (2004)" vs "extensão IA/Supabase (2026)". Modernizações cobrem 3 vetores:

1. **LLMs como dependências legítimas** (sem precedente em 2004) — fakear OpenAI/Anthropic clients, RAG retrieval como seam testável, prompts como código legacy que precisa de characterization
2. **Supabase como API-only application platform** (cap 15 modernizado) — Edge Functions wrappando Stripe/OpenAI/etc são exatamente "applications that are all API calls"; adapter/anti-corruption layer aplicado
3. **IA generativa como ferramenta de refactoring** — embeddings para detecção semântica de duplicação (cap 21 shotgun surgery modernizado), LLM para mental model + storytelling (cap 17 modernizado), LLM-generated mutants comportamentais (modernização de mutation testing tradicional)

**Beneficiários principais:**
- Suíte Supabase v1.8 (`supabase-edge-fn-writer` ganha API-only adapter pattern + payload capture)
- Suíte Observabilidade v1.9 (`omm-auditor` Capacidade 1 consulta legacy coverage; novo `/auditar-observabilidade-cobertura` complementa OMM)
- Suíte SRE v1.10 (`prr-conductor` Axe 5 consume REFACTOR-SAFETY; `/postmortem` consulta para regression cases)
- Suíte SRE v1.11 (load-shedding pattern aplicado em sprout/wrap quando legado tem cascading risk)
- Fluxo framework (todos os commands principais ganham awareness de legacy code)

### Fase 57: Skill — `legacy-api-only-applications` (cap 15 + Supabase Edge Functions)

**Status:** ⏳ pending
**REQs cobertos:** SKFD-LGC-09
**Artefato:** `kit/skills/legacy-api-only-applications/SKILL.md`
**Conteúdo (cap 15 original):** Applications que são quase 100% wrappers de APIs externas. Estratégia: extract adapter / anti-corruption layer; depende de uma interface mínima (não da API completa). Adapter testável; produção encanta para a API real.
**Modernização (2026):** Supabase Edge Functions wrappando Stripe/OpenAI/Twilio/etc são exatamente o caso paradigmático. Pattern canônico: `interface PaymentGateway → StripeAdapter / FakePaymentGateway → handler depende de interface`. Aplicado também a LLM clients (`interface LLMProvider → OpenAIAdapter / AnthropicAdapter / FakeLLMProvider`).

### Fase 58: Skill — `legacy-shotgun-surgery` (cap 21 + IA semantic search)

**Status:** ⏳ pending
**REQs cobertos:** SKFD-LGC-10
**Artefato:** `kit/skills/legacy-shotgun-surgery/SKILL.md`
**Conteúdo (cap 21 original):** "Mesma mudança em N lugares" — sintoma de duplicação estrutural. Detect via grep, extract method/function para 1 lugar, refactor incremental.
**Modernização (2026):** Detecção via **semantic search com embeddings** — `text-embedding-3-small` ou `pgvector` queries. Encontra duplicação ESTRUTURAL (mesmo if/else 5x) E duplicação SEMÂNTICA (mesma intenção em 3 implementações diferentes). Modernização sem precedente em 2004: regex pega "computeTotalCents" repetido, embeddings pegam também `calc_total_in_cents`/`getTotalInCents`/`totalCalculation`.

### Fase 59: Skill — `legacy-storytelling-naked-crc` (cap 16-17 + IA mental model)

**Status:** ⏳ pending
**REQs cobertos:** SKFD-LGC-11
**Artefato:** `kit/skills/legacy-storytelling-naked-crc/SKILL.md`
**Conteúdo (caps 16-17 originais):** "Telling the Story" — descreva o que o código faz em poucas frases para descobrir abstrações ausentes; Naked CRC — class-responsibility-collaborator sketches; estrutura emerge do exercício.
**Modernização (2026):** **IA produz primeiro draft do storytelling** — LLM lê arquivo/módulo, gera narrativa em 3-5 sentenças, identifica colaboradores principais, propõe sketch CRC. Humano valida e refina. 30 min de trabalho vira 5 min. Sem precedente em 2004 — então engenheiro tinha que ler tudo manualmente.

### Fase 60: Skill — `ai-prompt-characterization` (modernização total)

**Status:** ⏳ pending
**REQs cobertos:** SKFD-LGC-12
**Artefato:** `kit/skills/ai-prompt-characterization/SKILL.md`
**Conteúdo (sem precedente em 2004):** Prompts de produção e tools (function calling) são **legacy code também** — frequentemente 200-500 linhas, sem testes, mudanças quebram silenciosa. Pattern: characterization de generations LLM com `temperature=0` + `seed` fixo (Anthropic + OpenAI suportam) → outputs determinísticos → snapshot tests. Sanitização específica (normalizar timestamps, IDs gerados, tokens). Behavioral coverage = % de inputs cobrindo intents distintas.

Estratégia: (a) tokenizar funções/blocos coesos do projeto, (b) gerar embedding via `text-embedding-3-small` ou pgvector self-hosted, (c) clusterizar por similaridade (threshold ≥ 0.85), (d) reportar clusters como candidates a extract method/class, (e) ranquear por size × frequency × custo de extract. Output: `.planning/SHOTGUN-SURGERY.md`.
**Sem precedente em 2004:** detecção semântica era impossível sem ML maduro.

### Fase 67: Agente — `ai-mutation-tester` (modernização)

**Status:** ⏳ pending
**REQs cobertos:** AGCORE-LGC-07
**Artefato:** `kit/agents/ai-mutation-tester.md`
**Conteúdo:** Mutation testing tradicional usa mutants sintáticos (operador != → ==, valor 0 → 1). Esse agent usa LLM para gerar **mutants comportamentais** mais ricos: "remova esta validação", "inverta a ordem das chamadas a/b", "use o usuário do request em vez do auth.uid()". Cada mutant é semanticamente plausível mas comportamentalmente diferente. Roda suite contra cada mutant; survived mutants = pontos cegos no characterization.
**Sem precedente em 2004:** mutation testing era acadêmico; LLM-generated mutants é literatura recente (2023+).
