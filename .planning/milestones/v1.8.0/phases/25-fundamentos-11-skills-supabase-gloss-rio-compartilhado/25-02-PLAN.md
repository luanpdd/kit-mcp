### Task 2.1 — `supabase-migrations/SKILL.md`

**Path:** `D:/projetos/opensource/mcp/kit/skills/supabase-migrations/SKILL.md`

**Frontmatter:**
- `name: supabase-migrations`
- `description: Use ao criar arquivos de migration Supabase — naming YYYYMMDDHHmmss_short.sql, header de metadados, RLS obrigatório em toda nova tabela, policies granulares.` (≤ 200 chars)

**Must-include (verbatim):** `YYYYMMDDHHmmss`, `RLS`, `granular policies`

**Conteúdo concreto obrigatório:**
- Regra: naming canônico `YYYYMMDDHHmmss_short_description.sql` em UTC (ex: `20260506120000_create_tasks.sql`)
- Regra: header de comentário no topo de cada migration:
  ```sql
  /*
    Migration: create_tasks
    Created: 2026-05-06
    Purpose: Cria tabela tasks com RLS habilitado.
    Affects: public.tasks (new), public.tasks policies (new)
  */
  ```
- Regra: lowercase em todo SQL (alinhado com `supabase-postgres-style`)
- Regra: comentários copiosos em comandos destrutivos (`drop table`, `alter table ... drop column`, `truncate`)
- Regra: **RLS obrigatório** em toda nova tabela criada — `alter table ... enable row level security;` no mesmo arquivo
- Regra: policies **granulares** (uma `for select`, uma `for insert`, etc.) — NUNCA `for all` (ver `supabase-rls-policies`)
- Regra: `auth.uid()` SEMPRE envolvido em `(select auth.uid())` (ver `supabase-rls-policies` REGRA #1)
- Pattern canônico (code block EN):
  ```sql
  -- 20260506120000_create_tasks.sql
  create table public.tasks (
    id uuid primary key default gen_random_uuid(),
    user_id uuid not null references auth.users(id),
    title text not null,
    created_at timestamptz default now()
  );

  alter table public.tasks enable row level security;

  -- granular policies (one per operation)
  create policy "select_own_tasks"
    on public.tasks for select to authenticated
    using ((select auth.uid()) = user_id);

  create policy "insert_own_tasks"
    on public.tasks for insert to authenticated
    with check ((select auth.uid()) = user_id);

  create index on public.tasks (user_id);
  ```
- Anti-pattern: criar tabela sem `enable row level security` → tabela exposta a anon
- Anti-pattern: `for all` policy → granular policies por operação
- Anti-pattern: `drop column` sem comentário → comentário obrigatório com motivo + impacto

### Task 2.2 — `supabase-declarative-schema/SKILL.md`

**Path:** `D:/projetos/opensource/mcp/kit/skills/supabase-declarative-schema/SKILL.md`

**Frontmatter:**
- `name: supabase-declarative-schema`
- `description: Use ao gerenciar schema via supabase/schemas/ — workflow stop → db diff -f → revisar → apply. Inclui caveats sobre views, RLS, partitions.` (≤ 200 chars)

**Must-include (verbatim):** `supabase/schemas/`, `supabase stop`, `supabase db diff -f`, `caveats`

**Conteúdo concreto obrigatório:**
- Regra: workflow canônico — editar arquivos `.sql` em `supabase/schemas/` → rodar `supabase stop` (necessário antes de diff) → rodar `supabase db diff -f <name>` → **revisar** migration gerada → `supabase db reset` para aplicar
- Regra: nunca pular `supabase stop` antes de `db diff` — diff sem stop produz output inconsistente
- Regra: nunca pular **revisão manual** da migration gerada — diff é heurístico, pode gerar SQL incorreto
- Regra: DML (INSERT/UPDATE/DELETE) **não rastreável** por declarative — fica em migrations imperativas
- Section `## Caveats`:
  - **DML** (seeds, updates) → não rastreável por declarative; usar `supabase/seed.sql` ou migration imperativa
  - **View ownership** → diff não captura mudanças de owner
  - **RLS policies** → diff funciona mas alguns edge cases (policies que dependem de funções) precisam revisão extra
  - **Partitions** → suporte limitado; partitioned tables podem precisar migration manual
  - **Comments on objects** → diff captura mas verificar
  - **Alter publication** → não detectado por diff
  - **Create domain** → suportado mas verificar ordering
- Pattern canônico (code block EN):
  ```bash
  # PT-BR: workflow declarative — sempre stop antes de diff
  supabase stop
  supabase db diff -f add_tasks_index
  # PT-BR: revisar arquivo gerado em supabase/migrations/<timestamp>_add_tasks_index.sql
  supabase db reset
  ```
- Anti-pattern: rodar `supabase db diff` com containers up → output inconsistente; sempre `supabase stop` antes
- Anti-pattern: aplicar migration gerada sem revisão → diff pode gerar `drop column` indevido em renames

### Task 2.3 — `supabase-auth-ssr/SKILL.md`

**Path:** `D:/projetos/opensource/mcp/kit/skills/supabase-auth-ssr/SKILL.md`

**Frontmatter:**
- `name: supabase-auth-ssr`
- `description: Use ao bootstrap Next.js v16 + Supabase Auth — @supabase/ssr, getAll/setAll APENAS, NUNCA auth-helpers-nextjs, proxy completo com getUser e redirects.` (≤ 200 chars)

**Must-include (verbatim):** `getAll`, `setAll`, `NEVER use auth-helpers-nextjs`, `@supabase/ssr`

**Conteúdo concreto obrigatório:**
- **WARNING absoluto** no topo de Regras absolutas: `NEVER use auth-helpers-nextjs` — pacote DEPRECATED. Sempre `@supabase/ssr`
- Regra: padrão exclusivo `getAll`/`setAll` para cookies — NUNCA `get`/`set`/`remove` individuais (broken em Next.js v16+)
- Regra: separar browser client (`createBrowserClient`) de server client (`createServerClient`) — clientes diferentes para contextos diferentes
- Regra: proxy/middleware **completo** com `supabase.auth.getUser()` + redirects para login se sem user
- Regra: **auth method order** — após `createServerClient` mas ANTES de `getUser()`, NÃO chamar nada que produza response (cookies precisam fluir)
- Pattern canônico — browser client (code block EN):
  ```ts
  // utils/supabase/client.ts — PT-BR: client para Client Components
  import { createBrowserClient } from '@supabase/ssr'

  export function createClient() {
    return createBrowserClient(
      process.env.NEXT_PUBLIC_SUPABASE_URL!,
      process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!
    )
  }
  ```
- Pattern canônico — server client com getAll/setAll:
  ```ts
  // utils/supabase/server.ts — PT-BR: client para Server Components/Actions
  import { createServerClient } from '@supabase/ssr'
  import { cookies } from 'next/headers'

  export async function createClient() {
    const cookieStore = await cookies()
    return createServerClient(
      process.env.NEXT_PUBLIC_SUPABASE_URL!,
      process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
      {
        cookies: {
          getAll() { return cookieStore.getAll() },
          setAll(cookiesToSet) {
            try {
              cookiesToSet.forEach(({ name, value, options }) =>
                cookieStore.set(name, value, options))
            } catch {}
          }
        }
      }
    )
  }
  ```
- Pattern canônico — middleware com getUser + redirect:
  ```ts
  // middleware.ts — PT-BR: proxy obrigatório com getUser
  import { createServerClient } from '@supabase/ssr'
  import { NextResponse, type NextRequest } from 'next/server'

  export async function middleware(request: NextRequest) {
    let response = NextResponse.next({ request })
    const supabase = createServerClient(/* ... cookies getAll/setAll */)
    const { data: { user } } = await supabase.auth.getUser()
    if (!user && !request.nextUrl.pathname.startsWith('/login')) {
      return NextResponse.redirect(new URL('/login', request.url))
    }
    return response
  }
  ```
- Anti-pattern (Errado/Por quê/Certo):
  - **Errado:** `import { createMiddlewareClient } from '@supabase/auth-helpers-nextjs'`
  - **Por quê:** `auth-helpers-nextjs` é DEPRECATED + quebra em Next.js v16+ (cookies API mudou)
  - **Certo:** `import { createServerClient } from '@supabase/ssr'` com `getAll`/`setAll`
- Anti-pattern: `cookies: { get(), set(), remove() }` → `cookies: { getAll(), setAll() }`
- Anti-pattern: `NEXT_PUBLIC_SUPABASE_SERVICE_ROLE_KEY` em env → service_role NUNCA prefixado `NEXT_PUBLIC_*` (vaza para cliente — anti-pitfall B6)

### Task 2.4 — `supabase-realtime/SKILL.md`

**Path:** `D:/projetos/opensource/mcp/kit/skills/supabase-realtime/SKILL.md`

**Frontmatter:**
- `name: supabase-realtime`
- `description: Use ao implementar Realtime — broadcast com private:true, naming scope:entity:id, RLS sobre realtime.messages, removeChannel cleanup, migrar de postgres_changes.` (≤ 200 chars)

**Must-include (verbatim):** `broadcast`, `private: true`, `realtime.broadcast_changes`, `removeChannel`

**Conteúdo concreto obrigatório:**
- Regra: usar `broadcast` por padrão — `postgres_changes` é pattern legado, migrar
- Regra: **SEMPRE** `private: true` em canais novos — exige autenticação + RLS
- Regra: naming canônico `scope:entity:id` (ex: `room:messages:abc123`, `user:notifications:xyz789`)
- Regra: nomes de eventos `entity_action` (ex: `message_inserted`, `task_updated`)
- Regra: cleanup obrigatório com `supabase.removeChannel(channel)` em `useEffect return` (anti-pitfall B1)
- Regra: state checking antes de subscribe (`channel.state !== 'joined'`)
- Regra: error handling automático — Realtime tem retry built-in, mas log `status` no callback do `subscribe`
- Pattern canônico — client subscribe (code block EN):
  ```ts
  // PT-BR: subscrição com cleanup obrigatório
  useEffect(() => {
    const channel = supabase
      .channel('room:messages:' + roomId, { config: { private: true } })
      .on('broadcast', { event: 'message_inserted' }, (payload) => {
        setMessages((prev) => [...prev, payload.payload])
      })
      .subscribe((status) => {
        if (status === 'SUBSCRIBED') console.log('joined')
      })

    return () => {
      supabase.removeChannel(channel)
    }
  }, [roomId])
  ```
- Pattern canônico — RLS sobre `realtime.messages` (SELECT + INSERT separados, com index):
  ```sql
  -- PT-BR: SELECT policy permite ouvir mensagens do canal
  create policy "auth_select_messages"
    on realtime.messages for select to authenticated
    using ((select auth.uid()) is not null);

  -- PT-BR: INSERT policy permite enviar broadcast
  create policy "auth_insert_messages"
    on realtime.messages for insert to authenticated
    with check ((select auth.uid()) is not null);

  create index on realtime.messages (extension);
  ```
- Pattern canônico — DB trigger via `realtime.broadcast_changes`:
  ```sql
  create or replace function public.notify_message_insert()
  returns trigger language plpgsql security invoker set search_path = '' as $$
  begin
    perform realtime.broadcast_changes(
      'room:messages:' || new.room_id::text,
      'message_inserted', 'INSERT', 'messages', 'public', new, null
    );
    return new;
  end;
  $$;
  ```
- Anti-pattern (Errado/Por quê/Certo):
  - **Errado:** `supabase.channel('messages')` (sem `private`)
  - **Por quê:** canal público — qualquer cliente recebe payload sem RLS
  - **Certo:** `supabase.channel('room:messages:'+id, { config: { private: true } })`
- Anti-pattern: subscribe sem `removeChannel` no cleanup → memory leak + stale state
- Anti-pattern: `postgres_changes` em features novas → usar `broadcast` + trigger `realtime.broadcast_changes`
