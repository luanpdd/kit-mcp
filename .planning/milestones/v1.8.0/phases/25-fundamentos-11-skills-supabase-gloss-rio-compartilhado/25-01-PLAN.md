### Task 1.1 — `supabase-postgres-style/SKILL.md`

**Path:** `D:/projetos/opensource/mcp/kit/skills/supabase-postgres-style/SKILL.md`

**Frontmatter:**
- `name: supabase-postgres-style`
- `description: Use ao escrever SQL para Postgres/Supabase — snake_case, lowercase reserved, plurais para tabelas e singular para colunas, ISO 8601, CTEs lineares.` (≤ 200 chars)

**Sections (template D-03):** `## Quando usar` → `## Regras absolutas` → `## Patterns canônicos` → `## Anti-patterns` → `## Ver também`

**Must-include (verbatim):** `snake_case`, `ISO 8601`, `lowercase reserved`

**Conteúdo concreto obrigatório:**
- Regra: lowercase para palavras reservadas (`select`, `from`, `where` — nunca `SELECT`, `FROM`)
- Regra: `snake_case` para tabelas/colunas/funções; nunca camelCase ou PascalCase
- Regra: tabelas em **plural** (`books`, `authors`); colunas em **singular** (`title`, `author_id`)
- Regra: datas/horários em `ISO 8601` (`2026-05-06T12:00:00Z`)
- Regra: comentários `/* block */` para queries complexas + `-- line` para anotações curtas
- Regra: aliases descritivos com `as` explícito (`select b.title as book_title from books as b`)
- Regra: CTEs lineares para queries complexas — preferir múltiplas CTEs sobre subqueries aninhadas
- Regra: JOIN sempre com nomes completos da tabela qualificadora (`books.author_id = authors.id`)
- Anti-pattern: `SELECT * FROM Books WHERE Title='X'` → `select * from books where title = 'X'`
- Anti-pattern: `createdAt timestamp` → `created_at timestamptz`

### Task 1.2 — `supabase-rls-policies/SKILL.md`

**Path:** `D:/projetos/opensource/mcp/kit/skills/supabase-rls-policies/SKILL.md`

**Frontmatter:**
- `name: supabase-rls-policies`
- `description: Use ao criar/auditar RLS — sempre (select auth.uid()), policies separadas por operação, índices nas colunas, NUNCA user_metadata em autorização.` (≤ 200 chars)

**Must-include (verbatim):** `(select auth.uid())`, `WARNING user_metadata`, `TO authenticated`

**Conteúdo concreto obrigatório:**
- **REGRA #1 absoluta** (em maiúscula): SEMPRE envolver `auth.uid()` em `(select ...)` — sem o wrapper, queries com filtro RLS sofrem **1000× degradação** de performance
- **WARNING explícito** (em maiúscula no topo de Regras absolutas): `WARNING user_metadata` — NUNCA referenciar `auth.jwt() -> 'user_metadata'` em policy de autorização (cliente pode auto-elevar privilégio — privilege escalation B5). Usar `app_metadata` para roles administrativos
- Regra: policies separadas por operação (uma `for select`, uma `for insert`, uma `for update`, uma `for delete`) — NUNCA `for all`
- Regra: especificar role explicitamente com `to authenticated` ou `to anon` — nunca deixar implícito
- Regra: criar índice nas colunas referenciadas pela policy (`create index on tasks (user_id)`) — sem index, scan full
- Regra: MFA enforcement via `(auth.jwt()->>'aal')::text = 'aal2'` quando necessário
- Pattern canônico (code block EN):
  ```sql
  -- política de SELECT com wrapper obrigatório
  create policy "users_select_own_tasks"
    on tasks
    for select
    to authenticated
    using ((select auth.uid()) = user_id);
  ```
- Anti-pattern (Errado/Por quê/Certo):
  - **Errado:** `using (auth.uid() = user_id)` — sem `select`, executa por linha (1000× mais lento)
  - **Por quê:** Postgres reavalia `auth.uid()` por linha; o wrapper `(select)` permite Postgres executar uma vez e reusar
  - **Certo:** `using ((select auth.uid()) = user_id)`
- Anti-pattern (Errado/Por quê/Certo):
  - **Errado:** `using ((auth.jwt()->'user_metadata'->>'role') = 'admin')` — privilege escalation
  - **Por quê:** `user_metadata` é editável pelo próprio cliente via `auth.updateUser({data: {role: 'admin'}})`
  - **Certo:** usar `app_metadata` (set apenas via service_role + admin API) — `using ((auth.jwt()->'app_metadata'->>'role') = 'admin')`
- Anti-pattern: `for all` cobrindo CRUD inteiro → policies granulares por operação por role

### Task 1.3 — `supabase-database-functions/SKILL.md`

**Path:** `D:/projetos/opensource/mcp/kit/skills/supabase-database-functions/SKILL.md`

**Frontmatter:**
- `name: supabase-database-functions`
- `description: Use ao criar funções Postgres — SECURITY INVOKER por padrão, SET search_path = '' SEMPRE, schema-qualified names, IMMUTABLE/STABLE quando possível.` (≤ 200 chars)

**Must-include (verbatim):** `set search_path = ''`, `SECURITY INVOKER`, `schema-qualified`

**Conteúdo concreto obrigatório:**
- Regra: `SECURITY INVOKER` é **default obrigatório** — `SECURITY DEFINER` apenas com justificativa explícita (lint advisor 0011)
- Regra: `set search_path = ''` SEMPRE — sem isso, função vulnerável a hijack via `search_path` manipulation (anti-pitfall B7)
- Regra: schema-qualified names em todas as referências (`public.tasks`, não `tasks`) — sem qualifier, lookup falha quando `search_path = ''`
- Regra: marcar `IMMUTABLE` ou `STABLE` quando função não modifica dados (permite Postgres cachear resultado)
- Regra: error handling com `RAISE EXCEPTION 'message'` — nunca silent fail
- Pattern canônico (code block EN):
  ```sql
  create or replace function public.get_user_task_count(p_user_id uuid)
  returns integer
  language plpgsql
  security invoker
  set search_path = ''
  stable
  as $$
  declare
    v_count integer;
  begin
    select count(*) into v_count
      from public.tasks
      where user_id = p_user_id;
    return v_count;
  end;
  $$;
  ```
- Anti-pattern (Errado/Por quê/Certo):
  - **Errado:** `create function f() returns void security definer as $$ select count(*) from tasks; $$;` — sem `set search_path` + sem schema qualifier
  - **Por quê:** atacante pode criar `tasks` em schema próprio + manipular `search_path` para hijack
  - **Certo:** `security invoker set search_path = '' ... select count(*) from public.tasks`
