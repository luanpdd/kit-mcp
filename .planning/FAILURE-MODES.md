# FAILURE-MODES — kit-mcp

> Top-down catalog of how kit-mcp can fail, ordered by **impact × likelihood**, with current mitigation and follow-up. Companion to [`RUNBOOK.md`](./RUNBOOK.md) (which tells you what to *do* when one fires) and to the SLOs at [`./slos/`](./slos/) (which tell you whether the service is *officially* degraded).
>
> Pattern from skill [`production-readiness-review`](../kit/skills/production-readiness-review/SKILL.md), Operations + Security axes.
>
> Generated by Phase 96 (v1.18 — Eat Your Own Dog Food). Last reviewed 2026-05-09 (kit-mcp v1.17.0 baseline).

## How to read this table

- **Impact:** what the user/operator experiences if this fires unmitigated. `critical` = data integrity loss or arbitrary code execution. `high` = service unavailable / wrong output. `medium` = degraded UX. `low` = cosmetic.
- **Likelihood:** prior probability the failure fires *given current mitigations*. `blocked` (by-design — would require breaking a structural invariant), `low` (defended in depth, unlikely), `medium` (possible under adversarial conditions or rare configurations), `high` (expected at some non-zero rate in the field).
- **Current mitigation:** the specific phase/code path that prevents this today. Empty cell = no mitigation, raise alarm.
- **Follow-up:** known improvement queued on the roadmap, or `none` if the current state is the target.

## Failure-mode matrix

| # | Failure mode | Impact | Likelihood | Current mitigation | Follow-up | Runbook |
|---|---|---|---|---|---|---|
| 1 | MCP server boot hang on stdio (handshake never completes) | high | low | Phase 84.01 added a 5s init timeout + envelope-based error reporting; Phase 90.01 added `verifyManifest` cache so init isn't disk-bound on every spawn | None — current behavior is acceptable | [Scenario 1](./RUNBOOK.md#1-mcp-server-boot-fail) |
| 2 | Sidecar TCP race — hook fires but parent process exits before TCP write flushes; UI loses event | high | low | Phase v1.12.1 fix (commit `56b327f`) reordered `flush + exit` so TCP write completes synchronously; SEC-13-05 invariant for the sidecar bind cap | None — fix is in v1.12.1+ | [Scenario 2](./RUNBOOK.md#2-sidecar-ui-hangunresponsive) |
| 3 | Arbitrary code execution via `gates.run` over MCP transport | critical | blocked-by-design | Phase 79.01: `gates.run` requires `KIT_MCP_ALLOW_GATES=1` to be set in the server's environment, AND only well-known gate IDs from `kit/gates/` are dispatchable. The MCP server *cannot* execute arbitrary shell — that surface does not exist in the dispatcher | None — guard is structural | n/a |
| 4 | Path traversal during `kit sync install` writes outside the project root | critical | blocked-by-design | Phase 79.02 + SEC-14-03: every destination path is resolved against the canonicalized project root and rejected if it escapes. Symbolic links in the kit tree are dereferenced and re-validated | None — guard is structural | [Scenario 5](./RUNBOOK.md#5-sync-corruption-partial-write) |
| 5 | Manifest tampering — published tarball contains kit content that diverges from the signed manifest | high | blocked | Phase 83.03: `verifyManifest` reads `kit/MANIFEST.json` and SHA-256-checks every file. Mismatch returns `EMANIFESTMISMATCH` *before* sync writes anything. Phase 84.01 preserves the error code through the sanitized envelope so MCP clients see the failure clearly | Bumping manifest to per-file BLAKE3 once Node ships native support — no urgency | [Scenario 3](./RUNBOOK.md#3-manifest-mismatch-on-sync) |
| 6 | Token leak in MCP error envelope (API key, OAuth secret, or absolute path leaked to LLM client) | high | medium | Phase 84.01 introduced `redactSecrets` — strips `e.stack`, redacts `/Users/<u>/...` and `/home/<u>/...` paths to `<HOME>/...`, masks `sk-...`, `Bearer <jwt>`, and `eyJ...` JWT-shaped strings. Preserves `err.code` for triage | Wider regex coverage as new credential formats surface (e.g. fine-grained PATs) | [Scenario 4](./RUNBOOK.md#4-npm-publish-workflow-fail) when audit gate flags a leaked token in deps |
| 7 | Sync wall time regression — large kits take seconds, not the ~400ms baseline | low | medium | Phase 88.01 (`BATCH_SIZE=16` concurrent IO) + Phase 91.01 (diff filter on `treeCopy` ops, mtime+size heuristic). Steady-state re-sync is ~390ms for the bundled 321-file kit on a developer laptop (see [BENCHMARK.md](./BENCHMARK.md)) | Watching: a personal kit with 1000+ files is the natural regression boundary. Add a perf gate test if a user reports >2s | [Scenario 5](./RUNBOOK.md#5-sync-corruption-partial-write) only if regression masks a write failure |
| 8 | Cold start regression — `kit list-agents --terse` exceeds ~300ms median on a typical dev laptop | low | medium | Phase 89.01 (lazy imports for optional deps) + Phase 97 (planned coverage ratchet, no perf impact expected). Baseline 232ms median on the v1.17.0 reference machine (see [BENCHMARK.md](./BENCHMARK.md)) | Cold-start CI gate is candidate work; not yet wired | n/a — diagnose via [Scenario 1](./RUNBOOK.md#1-mcp-server-boot-fail) |
| 9 | npm publish workflow fails on tag push (red CI, no published artifact) | medium | medium | `npm audit` gate enforces clean dep tree on every release; CHANGELOG check; manifest regen on `prepublish`. Workflow is hermetic (no network in build, only on `npm publish` itself) | Migrate to provenance-attested publish once npm/GitHub Actions integration is GA | [Scenario 4](./RUNBOOK.md#4-npm-publish-workflow-fail) |
| 10 | `.planning/STATE.md` corruption — hand-edited or partially written by a crashed framework command | medium | low | Schema validation on read in `tools.cjs state` commands; commands exit non-zero on malformed YAML frontmatter rather than silently overwriting | Add a `state repair` subcommand (currently `state init` from-scratch only) | n/a — diagnose via doctor: `kit doctor --json` |
| 11 | Antivirus / Windows Defender quarantines `.md` files mid-sync (Windows-specific) | medium | low | None — kit-mcp surfaces the OS error (`EACCES`/`EPERM`) verbatim rather than swallowing it | Document the whitelist guidance in [Scenario 5](./RUNBOOK.md#5-sync-corruption-partial-write) — already done in the RUNBOOK | [Scenario 5](./RUNBOOK.md#5-sync-corruption-partial-write) |
| 12 | Disk full during sync (`ENOSPC`) leaves partial tree | medium | low | Phase 91.01 written[] return list lets a re-run fix exactly the missing files (mtime+size heuristic); sync is idempotent | Pre-flight free-space check is candidate work; current behavior surfaces error clearly | [Scenario 5](./RUNBOOK.md#5-sync-corruption-partial-write) |

## Risk-tier rollup

> Per [`sre-risk-management`](../kit/skills/sre-risk-management/SKILL.md), only the top tier needs an explicit budget call.

| Tier | Modes | Why this tier |
|---|---|---|
| Critical (blocked-by-design) | 3, 4 | Structural invariants the dispatcher and sync writer enforce. Any regression here is a fatal release blocker — not a budgetable risk. Phase 79 + SEC-14 cycle established the guards; Phase 84.01 protected the error path. |
| High impact, low likelihood | 1, 2, 5, 6 | Defended in depth with multiple phases of work. Re-budget annually. |
| Low/medium impact, medium likelihood | 7, 8, 9 | The "noise floor" of a developer-tool service. Tracked via SLOs; if MCP-tool latency p95 SLO trips, mode 7 or 8 is the candidate hypothesis. |
| Long-tail OS/environment | 10, 11, 12 | Acknowledged but mitigated by surfacing errors faithfully. Any one of these firing in production should produce a [`blameless-postmortems`](../kit/skills/blameless-postmortems/SKILL.md) entry, not a code change, until pattern justifies the investment. |

## Modes deliberately *not* on this list

A failure absent here is either:

- **Out of scope** — kit-mcp is a developer-tool MCP server, not a hosted service. Multi-tenant isolation, DDoS, geo-redundancy: all out of scope; the MCP server runs as a child of the user's IDE.
- **Already an existence proof** — e.g. "MCP transport itself fails": that's the SDK's responsibility (`@modelcontextprotocol/sdk`), and the IDE's recovery loop. We do not duplicate.
- **Budgeted under SLO** — e.g. one tool dispatch returning an error counts toward `mcp-tool-availability.yml`'s 0.5% monthly error budget. The SLO is the failure mode for that case; this list catalogs *modes outside the SLO envelope*.
