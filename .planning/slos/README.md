# SLOs — kit-mcp

> Service Level Objectives applying the canonical event-based pattern from the [`event-based-slos`](../../kit/skills/event-based-slos/SKILL.md) skill (chapter 12 of *Observability Engineering*, O'Reilly 2022) — and from chapter 4 of the Google SRE book.
>
> Generated by Phase 95.01 (v1.18 — Eat Your Own Dog Food).

## SLOs defined

| File | What | Target | Window | Source SLI |
|---|---|---|---|---|
| [`mcp-tool-availability.yml`](./mcp-tool-availability.yml) | Ratio of successful MCP tool dispatches | 99.5% | 30d sliding | `counters[*:ok] / (counters[*:ok] + counters[*:error])` |
| [`mcp-tool-latency.yml`](./mcp-tool-latency.yml) | p95 wall-clock latency of MCP tool dispatches | ≤ 200ms | 30d sliding | `percentile(histograms[tool], 95)` |

## How the SLI is calculated

Phase 94.01 (`OBS-18-01`/`OBS-18-02`) instrumented the kit-mcp server with in-memory golden signals — counters keyed `${tool}:${status}` and per-tool latency histograms (FIFO cap N=1000). The dispatcher in `src/mcp-server/index.js` wraps the central catch and emits one observation per dispatch on every path: success, thrown error, unknown-tool-name. Counters are pure in-memory Maps, histograms are bounded arrays sorted on snapshot.

The new MCP tool `metrics-snapshot` (parameterless, read-only) returns the live snapshot as JSON:

```json
{
  "counters": {
    "sync:ok": 142,
    "sync:error": 1,
    "mapeamento:ok": 38,
    "metrics-snapshot:ok": 7
  },
  "latency": {
    "sync":             { "p50": 45.2, "p95": 178.4, "p99": 234.1, "count": 143 },
    "mapeamento":       { "p50": 12.0, "p95":  28.7, "p99":  41.5, "count":  38 },
    "metrics-snapshot": { "p50":  0.4, "p95":   0.9, "p99":   1.2, "count":   7 }
  }
}
```

From a snapshot you derive the SLIs at read-time:

- **Availability** — sum `:ok` counters, sum `:ok + :error`, divide. Compare against `0.995`.
- **Latency** — read each tool's `latency.<tool>.p95`. Compare against `200`. SLO is met when ALL tools' p95 ≤ target (or per-tool, depending on consumer).

The SLI is **event-based, not time-based** — every dispatch is one event with a binary outcome (or one latency sample). The skill `event-based-slos` calls time-based SLIs (e.g., "% of 5-min windows under threshold") an anti-pattern: pre-aggregation hides outliers and one bad minute can dominate a percentile.

## Workflow — calculating SLI from running server

```bash
# 1. Spawn an MCP-aware client and emit a metrics-snapshot request.
#    The simplest in-repo way is the integration test pattern:
#    test/integration/mcp-metrics-snapshot.test.js

# 2. The kit-mcp `/burn-rate-status` command (already shipped in v1.10) reads
#    these YAML files plus a recent snapshot and renders burn-rate tables.
#    See `.claude/commands/burn-rate-status.md`.

# 3. For ad-hoc inspection during development, set the boot-time reset hook:
KIT_MCP_METRICS_RESET=1 node bin/mcp.js
#    Then drive a few requests through your MCP client and re-read snapshot.
```

## Why these SLOs and not others

The skill `event-based-slos` is opinionated:

- **Granular per journey, not global.** kit-mcp's "journey" is a tool dispatch, so we have one availability SLO and one latency SLO covering the dispatcher. We do NOT define per-tool SLOs today — the toolkit is small and per-tool error budgets would be too thin to be meaningful. When a single tool grows enough usage to merit its own SLO (e.g., `sync` for large kits), split it out.
- **No SLA-as-SLO.** kit-mcp has no external SLA; the SLO is purely an internal contract with the developer using the tool.
- **No 99.99% targets.** The risk continuum says four-nines is indistinguishable from five-nines on a smartphone+ISP path; the skill caps SLO targets at 99.95%. Anything tighter belongs in informational dashboards, not alertable SLOs.
